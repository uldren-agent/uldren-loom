# 0060 - FIPS Distribution and Compliance

**Status:** Draft
**Normative:** Mixed
**Capability:** `fips-distribution`

## 1. Purpose

Loom supports a default identity profile and a FIPS identity profile. This spec defines how that
runtime profile model combines with release-channel, TLS-provider, packaging, support, and compliance
requirements.

The goal is an honest enterprise compliance boundary:

- a standard Loom distribution for broad platform reach;
- a FIPS Loom distribution for regulated deployments;
- clear runtime policy so a FIPS-required Loom is not served through a non-FIPS TLS or provider path;
- supported migration from default-profile Looms into FIPS-profile Looms.

## 2. Distribution Channels

Loom has two release channels.

| Channel | Purpose | Default identity profile | TLS provider claim | Default strictness |
| --- | --- | --- | --- | --- |
| Standard `loom` | Broad platform reach, development, local tools, general users | `default` / BLAKE3 | No FIPS TLS claim | FIPS policy available only where source-backed |
| FIPS `loom` | Regulated customers, paid support, and compliance deployments | `fips` / SHA-256 | AWS-LC-FIPS through rustls `fips` | Strict FIPS policy |

The standard binary may create and open FIPS-profile Looms. It must not claim FIPS TLS unless it is
built and packaged with the FIPS TLS provider. If a Loom is marked FIPS-required, the standard binary
must not serve it over a non-FIPS listener.

The FIPS binary defaults new stores to the FIPS identity profile. In strict mode it rejects non-FIPS
stores except through explicit migration tooling.

## 2.1 Current implementation

Current source backs the data-profile side of this model and the first hosted runtime guards. A
standard binary can create and open profile-specific stores according to the existing digest/profile
rules. Served listener records now carry TLS mode, certificate/key references, and trust-bundle
references. `loom-hosted` exposes a source-backed runtime profile that reports the binary channel,
runtime policy, default identity profile, TLS provider label, FIPS capability, and whether the build
makes a FIPS TLS claim. The standard build reports `standard` / `capable`, defaults new stores to the
default BLAKE3 profile, reports no FIPS TLS claim, and rejects FIPS-required stores from hosted serving.
The `fips` feature selects the rustls `fips` feature path, reports `fips` / `strict`, defaults new
stores to the SHA-256 FIPS profile, and rejects non-FIPS stores from hosted serving. `loom init --fips`
rejects a conflicting explicit default identity profile. HTTP served listeners enforce the configured
request-size limit before request handling. Direct TLS is source-backed for the current TLS-enabled
hosted HTTP, CalDAV, CardDAV, IMAP, and JMAP listener runtimes through rustls with PEM
certificate/key loading and optional PEM trust-bundle loading for client-certificate verification.
Invalid TLS material fails closed with audited rejection, and HTTP listeners enforce the stored
`owner-or-passphrase` or `passphrase` auth mode. HTTP listeners run through a shared hosted runner
that enforces the stored request-size, idle-timeout, and session-timeout limits. Hosted external-auth
build profiles now select an explicit provider channel: standard served builds use the non-FIPS
AWS-LC provider feature, FIPS served builds use the AWS-LC-FIPS provider feature, and mixed provider
selection fails at compile time. FIPS hosted builds source-back direct public-key proof verification
through AWS-LC and mTLS binding to rustls-verified peer certificates. Direct OIDC, SAML, and passkey
proof verification is intentionally unsupported in FIPS builds; FIPS deployments must use
already-verified external assertions from a compliant identity provider or gateway for those
credential families. `loom store copy` is source-backed for profile-preserving file
copies, optional compaction, dry-run planning, and committed default-profile to FIPS-profile workspace
migration. The FIPS migration path rewrites
commit/tree/tag/chunk links, file content addresses, prolly internal links, and queue stream payload
references under the destination profile; copies identity and ACL metadata; imports audit
retention/legal-hold policy; imports served-listener intent as disabled listener records; rejects
dirty workspaces; and rejects encrypted profile-changing copies until an explicit encrypted-target
policy is added. `loom store copy --format text|json --report-file <path>` is source-backed for
formal JSON migration reports, including source and destination encryption status. Profile-changing
copies from encrypted default-profile stores now create encrypted FIPS-profile targets using
`--new-key-source`, preserving committed workspace data while changing both the store identity profile
and the destination unlock credential. `loom store policy --fips-required <true|false>` persists the
store-level compliance marker separately from the identity profile, and hosted listener startup uses
that marker when enforcing non-FIPS runtime rejection. Normal native Loom opens, OPFS-backed Loom
opens, high-level store creation helpers, C ABI key-management updates, and Node/Python direct
key-management helpers enforce the same runtime store policy. Native bindings expose provider/profile
reporting, WASM exposes compatibility profile reporting without a native FIPS certification claim, and
the memory-store conformance report serializes the linked runtime provider profile. `just
build-release-fips` builds the FIPS CLI channel, `just fips-check` compiles/tests the FIPS hosted
runtime path, and `scripts/release-materials.sh` captures Cargo metadata, dependency tree, toolchain
versions, source revision, binary SHA-256, checksums, an SPDX 2.3 JSON SBOM, and an unsigned signing
manifest. The current source does not yet implement actual release artifact signing/publishing, gRPC
direct TLS, generated capability resources, or published binding packages.

## 3. Runtime Policy

Runtime policy is separate from build identity.

| Runtime policy | Meaning |
| --- | --- |
| `off` | No FIPS restrictions beyond normal store/profile validation. |
| `capable` | The binary can operate FIPS-profile data, but compliance depends on the build channel and served-provider path. |
| `strict` | FIPS-required behavior is enforced. Non-FIPS stores, algorithms, and hosted listeners are rejected except through migration tooling. |

The FIPS binary starts in `strict` policy by default. The standard binary may expose `capable` behavior
for FIPS-profile data, but must report that its TLS/provider path is not FIPS unless the FIPS provider
is actually compiled and packaged.

Identity verifier choices are owned by 0026. For FIPS deployments, Loom-local direct OIDC, SAML, and
passkey proof verification is not a compliance surface today. Those credential families enter through
already-verified external assertions from a compliant identity provider or gateway. Public-key
challenge verification uses AWS-LC-backed primitives in FIPS builds, and mTLS identity binds to
rustls-verified peer certificate material.

## 4. Hosted Listener Rules

A hosted listener is compliant only if every layer in the path is compliant:

- the store is FIPS-profile when FIPS is required;
- the binary is a FIPS build;
- the TLS provider is AWS-LC-FIPS through rustls `fips`, or a future approved provider profile;
- the listener policy does not enable non-FIPS algorithms;
- the configured certificate/key and trust path meet deployment policy;
- conformance and capability reports identify the provider profile.

If a Loom is marked FIPS-required, serving it through a non-FIPS TLS or provider path is forbidden.
This includes REST, JSON-RPC, MCP HTTP, gRPC, and future hosted protocols.

Local non-network operations may be allowed by policy only when they do not make a FIPS compliance
claim. The FIPS binary strict mode should reject non-FIPS stores rather than silently downgrading.

## 5. Bindings

Language bindings do not get their own daemon or `loom serve` command. They link or load a Loom
runtime and inherit the runtime's crypto and profile behavior.

Bindings that embed Loom in a closed application environment, such as mobile or browser-style
containers, are responsible for application lifecycle and local file ownership. They are not expected
to manage the CLI daemon. If a binding package claims FIPS behavior, its native artifact must be built
from the FIPS channel and must report the same provider profile as the CLI.

Current native bindings expose a source-backed runtime provider/profile report as canonical CBOR:
Node `runtimeProfile()`, Python `runtime_profile()`, C++ `loom::runtime_profile()`, Swift/iOS
`Loom.runtimeProfile()`, JVM `Loom.runtimeProfile()`, Android/Kotlin `Loom.runtimeProfile()`, and
React Native `runtimeProfile()`. The report identifies binary channel, runtime policy, default
identity profile, crypto provider, TLS provider, FIPS capability, and FIPS TLS claim. WASM exposes
`runtime_profile()` with the same field shape as compatibility evidence for the browser artifact.
Browser runtimes still cannot claim native FIPS certification by loading this WASM artifact.

The current runtime store-policy guard is source-backed across normal native Loom open helpers,
OPFS-backed Loom open helpers, high-level create helpers, C ABI key-management updates, and
Node/Python direct key-management helpers. A non-FIPS-capable artifact rejects a store marked
FIPS-required. A standard artifact may still open FIPS-profile data when the store is not marked
FIPS-required and no FIPS compliance claim is made.

Therefore the first FIPS distribution gate is the CLI/server binary channel. Binding FIPS packages are
a separate packaging and certification surface, not part of `loom serve`. Current source backs the
first binding packaging evidence layer: Node, Python, and WASM binding crates expose a `fips` feature
that propagates to `loom-core`, and `scripts/binding-release-materials.sh` records binding package
names, build recipes, runtime-profile surfaces, FIPS claim eligibility, discovered package/native
artifacts, and SHA-256 checksums for standard and FIPS channels. WASM remains explicitly marked as
compatibility evidence only, not a native FIPS certification claim.

## 6. Certificates and Trust

Loom supports multiple trust sources.

| Trust source | Use |
| --- | --- |
| OS roots | Ordinary client verification where the host operating system provides a managed trust store. |
| Embedded root bundle | Portable client verification, especially when OS roots are unavailable or inconsistent. |
| Loom-managed trust bundle | Enterprise/private CA roots and pinned trust material stored as Loom control-plane state. |
| Operator-provided server certificate/key | Hosted TLS identity for REST, JSON-RPC, MCP HTTP, gRPC, and future protocols. |
| ACME-managed certificate | Future automation for public HTTPS deployments. |

No universal server certificate is embedded in Loom. A server certificate identifies a particular
deployment and must be supplied or managed for that deployment.

## 7. Compliance Artifacts

The FIPS release channel requires compliance artifacts beyond a successful build.

Required artifacts:

- provider profile and AWS-LC-FIPS module version;
- target platform and architecture matrix;
- SBOM for the binary and packaged dynamic libraries;
- installer/runtime validation steps, including dynamic library path checks where applicable;
- startup verification result for FIPS mode;
- conformance report showing identity profile, TLS provider, hosted listener policy, and source
  revision;
- release notes for crypto-provider updates and module-version changes.

Current binding package material capture is source-backed through `just binding-release-materials`
and `just binding-release-materials-fips`. `report_memory_store().to_json()` now also serializes
`release_certification` rows for standard and FIPS release materials, standard and FIPS binding
material capture, binding package compatibility metadata, binding package signing-manifest material,
binding package promotion policy, binding publication dry-run gate evidence, browser/device runtime
evidence, provider profile evidence, target publication/signing work, and unsupported WASM
native-FIPS claims. These commands and report rows do not publish packages, apply signatures, attach
attestations to registries, or choose alternate FIPS package names. They produce the evidence input a
release pipeline needs before publishing and signing.

## 8. Paid Support Boundary

FIPS support is a separate support boundary because it carries platform, packaging, compliance, and
incident-response obligations.

The paid support surface should include:

- supported operating systems and architectures;
- supported AWS-LC-FIPS module version;
- packaging guidance for dynamic libraries on macOS and Windows;
- upgrade policy for crypto modules;
- vulnerability notification and patch windows;
- customer-facing validation documentation;
- migration support from default-profile Looms.

The standard distribution should avoid making compliance claims that belong only to this supported
FIPS channel.

## 9. Migration

FIPS migration is a rekey/import operation into a new FIPS-profile Loom, not an in-place flip of object
identity.

Required migration behavior:

- read a default-profile Loom under normal policy;
- create a new FIPS-profile Loom;
- copy/import committed supported workspaces, facets, refs, principals, and ACLs;
- rewrite profile-sensitive object, content, prolly, and stream-record references;
- reject dirty workspaces rather than dropping uncommitted working-tree state;
- preserve audit retention/legal-hold policy;
- transform served-listener intent into disabled listener records that an operator must review before
  enabling;
- migrate encrypted default-profile sources into encrypted FIPS-profile targets with a destination
  credential supplied by `--new-key-source`;
- produce a migration report with source profile, destination profile, omitted items, and operator
  decisions.

The destination Loom may then be marked FIPS-required with `loom store policy --fips-required true`.
Once marked FIPS-required, hosted serving must enforce the rules in section 4.
