## How to execute a row

1. Open the row in `V2 plan worklist`.
2. Open the owning spec gate named by `Primary owner`.
3. Implement from the owning spec gate.
4. Use this file's matching V2 detail section only for dependency and boundary context.
5. Run the relevant checks.
6. Update both the owning spec gate and the row in `V2 plan worklist`.

Do not implement from `Reference - spec disposition matrix`, `Plan construction history`,
`Reference - findings that shaped the queue`, or the source evidence appendix. Those sections explain
why the queue is shaped the way it is.

## Status vocabulary

| Status | Meaning |
| --- | --- |
| Active - implementation owner | Pullable implementation work. The row owns remaining implementation, conformance, reporting, or certification work. |
| Closed - plan mapping | Closed plan-construction work. Do not treat this as implementation completion unless the row explicitly says so. |
| Source-backed | The checked source inputs identify an implementation and at least one evidence path. |
| Source-backed subset | A subset is implemented, but the indexed spec names remaining target work. |
| Mapped, not implementation-complete | The source-backed scope, target gaps, findings, and decision state are mapped, but implementation and conformance work remain open. |
| Needs source audit | The index or module inventory is not enough to classify completion. Source, tests, IDL, ABI, and conformance still need review. |
| Target | The index or conformance inventory describes planned work without checked implementation proof in the current assessment. |
| Blocked | Work cannot responsibly proceed until the owning spec or owner decision resolves the boundary. |
| Archived | Not part of the active implementation gate. Requires fresh promotion before work resumes. |

Completion state: indexed to owning specs, not implementation-complete. This section records the

Decision Points:

1. Question: What should the passkey descriptor shape be?
   Context: 0034 reserves passkey provider space but does not promote a wire or storage shape.
   Examples: Store credential id and PRF salt in `WrapEntry.params`, or require host-specific
   opaque provider references.
   Options: Use credential id plus PRF salt in `params`; use an opaque host handle; defer passkey
   promotion until a separate provider registry exists.
   Recommendation: Use credential id plus PRF salt in `params` because they are not secret and make
   passkey unlock deterministic across passkey managers.
   Consequence of deferring: Passkey rows must remain target/unsupported, and capability reports
   cannot advertise passkey unlock as a promoted provider.
2. Question: Should KMS/HSM KEKs be cached?
   Context: 0034 keeps provider acquisition outside `loom-core` and treats the raw KEK as a host
   supplied secret.
   Examples: No persisted cache, host-owned in-memory TTL cache, or persistent wrapped cache.
   Options: No cache; optional host-owned in-memory TTL cache; persistent cache under Loom metadata.
   Recommendation: No persisted cache and no default DEK cache; allow an optional host-owned
   in-memory TTL cache when deployment policy explicitly enables it.
   Consequence of deferring: KMS/HSM providers cannot be promoted beyond target rows because unlock
   latency, replay, revocation, and exposure semantics remain undefined.
3. Question: Should Loom keep the current pull model or add a callback ABI for provider unlock?
   Context: Current source implements caller-supplied passphrase bytes or caller-supplied 32-byte KEK
   bytes, and 0034 says provider SDKs should stay outside `loom-core`.
   Examples: Host obtains a KEK and calls `loom_open_with_kek`, or Loom calls a provider callback
   during open.
   Options: Keep pull acquisition; add a narrow callback ABI; add provider SDKs to core.
   Recommendation: Keep pull acquisition as the v1 promotion path and add a callback ABI only if a
   provider cannot work through host-side acquisition.
   Consequence of deferring: Provider integrations can still be prototyped in hosts, but the public
   provider contract and conformance cannot be promoted.
4. Question: How should FIPS/provider posture be represented?
   Context: 0034 says provider posture or attestation metadata is not implemented, while 0060 owns
   FIPS release boundaries.
   Examples: Informational posture in `params`, attestation evidence linked from admin reports, or a
   hard trust boundary inside key metadata.
   Options: Informational posture field; separate reviewed attestation extension; hard trust field in
   wrap metadata.
   Recommendation: Use an informational provider posture field in `params`; keep verified
   attestation as a later deployment-specific extension.
   Consequence of deferring: FIPS/provider rows must avoid claiming hardware-backed or attested
   unlock support, even where raw-KEK plumbing exists.
