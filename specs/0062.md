# 0062 - Inference Model Downloads

**Status:** Complete for scoped source-backed implementation; extension work delegated. **Version:** 0.2.1-source-backed.
**Capability:** `inference-downloads`.

**Depends on:** 0043 (provider surface), 0050 (embedding providers), 0051 (LLM / chat providers),
0017 (vector layer), 0032 (native / web parity), 0035 (durable delivery), 0036 (locking and
coordination). **Relates to:** `specs/ai/AI-0001-providers-binding.md`.

This spec defines how Loom discovers, downloads, configures, removes, diagnoses, and activates local
models for LLM chat/completion and text embedding.

The names are:

- `llm`: a chat/completion model used through `loom_core::inference::Llm`.
- `text-embedding`: a text embedding model used through `loom_core::inference::TextEmbedding`.

`Inference` is not the provider type name. "Inference" is the umbrella capability and crate name.

Implementation status: complete for the scoped `llm` and `text-embedding` acquisition and activation
path. The source tree has the `loom-inference` crate boundary, shared
descriptor types, native `hf-hub` acquisition, live shared-cache discovery for curated local models,
CLI model and instance management, default-on doctor inference and hardware reporting, curated model
catalog behavior, Candle CPU text-embedding activation, Candle CPU small-LLM activation, Ollama
adapter contracts, MLX bundle inspection, first-class vector text upsert/query CLI commands, and a
vector text embedding demo. DP1, DP2, DP3, and DP4 are decided in this document. Optional CUDA, MLX,
Core ML, hosted API, Ollama, and llama.cpp surfaces are represented by source-backed runtime flags,
adapter contracts, doctor checks, or explicit bridge-missing behavior where provider execution is not
promoted. Non-text vector source pipelines are outside 0062 and owned by 0068 before any image,
audio, video, PDF, or multimodal embedding model is promoted. Full workspace `just ci` is green for
this closure state.

## 2. Crate Boundary

Move the provider contracts and model acquisition surface out of `loom-core`.

Target crates:

```text
loom-types
  shared inference model descriptors, runtime descriptors, hardware report types,
  ephemeral download job status shapes, and stable CBOR/JSON shapes that must not depend on core

loom-inference
  Llm and TextEmbedding provider traits, local and remote provider adapters,
  Hugging Face / browser fetch model acquisition, live cache discovery, download manager,
  runtime compatibility checks, and doctor probes

loom-core
  depends on loom-inference; re-exports loom_core::inference::{Llm, TextEmbedding}
  for callers that already import inference through core
```

Rules:

- `loom-inference` MUST NOT depend on `loom-core`.
- `loom-inference` MAY depend on `loom-types`.
- `loom-core` MAY depend on `loom-inference`.
- Shared model and ephemeral job status types that cross crate boundaries live in `loom-types`.
- `loom-core` keeps no concrete downloader, tokenizer, or model runtime.
- The vector facet continues to depend on a text embedding handle, but that handle comes from
  `loom-inference`.

The public provider names are:

```rust
pub trait Llm: Send + Sync {
    fn model_id(&self) -> &str;
    fn complete(&self, request: &LlmRequest) -> Result<LlmResponse>;
    fn stream(&self, request: &LlmRequest) -> Result<Box<dyn LlmStream>>;
}

pub trait TextEmbedding: Send + Sync {
    fn model_id(&self) -> &str;
    fn dimension(&self) -> usize;
    fn weights_digest(&self) -> Option<&str>;
    fn embed(&self, texts: &[String]) -> Result<Vec<Vec<f32>>>;
}
```

Migration from current names:

| Current source name | Target name | Compatibility path |
| --- | --- | --- |
| `InferenceProvider` | `Llm` | Re-export or type alias for one release window if needed. |
| `Inference` | `LlmHandle` or direct `Box<dyn Llm>` host handle | Do not expose `Inference` as the final public type. |
| `EmbeddingProvider` | `TextEmbedding` | Re-export or type alias for one release window if needed. |
| `Embeddings` | `TextEmbeddingHandle` or direct `Box<dyn TextEmbedding>` host handle | Keep vector-facing behavior but rename. |
| `EmbeddingModel` | `TextEmbeddingModel` | `EmbeddingModel` may remain as a compatibility alias. |

## 3. Model Descriptor Types

`loom-types` owns the shared descriptor layer.

```rust
pub enum InferenceModelKind {
    Llm,
    TextEmbedding,
}

pub enum InferenceRuntimeKind {
    CandleSafetensors,
    CandleGguf,
    Mlx,
    Ollama,
    LlamaCpp,
    OpenAiCompatibleHttp,
    HostedApi,
}

pub struct InferenceModelDescriptor {
    pub model_id: String,
    pub kind: InferenceModelKind,
    pub source: InferenceModelSource,
    pub revision: Option<String>,
    pub files: Vec<InferenceModelFile>,
    pub runtimes: Vec<InferenceRuntimeDescriptor>,
    pub embedding_dimension: Option<usize>,
    pub context_window: Option<u32>,
    pub weights_digest: Option<String>,
}
```

The descriptor is metadata. It does not download, load, tokenize, run, or authorize a model.

Required behavior:

- `InferenceModelKind::Llm` resolves to an `Llm`.
- `InferenceModelKind::TextEmbedding` resolves to a `TextEmbedding`.
- A downloaded text embedding model can produce the vector-facing model profile.
- A downloaded LLM model can produce a model id for responses and cached derived artifacts.
- Runtimes are explicit and testable. A model being downloadable from Hugging Face does not mean it is
  runnable by candle, MLX, Ollama, or llama.cpp.

## 4. Download Library Measurement

DP1 is evidence-driven. The first measurement is source-backed in `prototypes/size-probes` as of this
revision.

Probe setup:

- release build;
- stripped binary via the probe crate release profile;
- `lto = "thin"`;
- one feature active at a time;
- baseline empty binary: 338,992 bytes.

Measured on 2026-07-05:

| Probe | Feature | Size | Delta | Normal deps | Finding |
| --- | --- | ---: | ---: | ---: | --- |
| baseline | `baseline` | 338,992 B | 0 B | 0 | Empty control. |
| `hf-hub` defaults | `hf_hub_default` | 2,531,984 B | +2,192,992 B | 181 | Pulls async and sync clients plus native TLS. Too broad. |
| `hf-hub` async rustls | `hf_hub_tokio_rustls` | 2,397,600 B | +2,058,608 B | 157 | Rustls-only but async-only. Good if CLI already hosts Tokio. |
| `hf-hub` blocking ureq | `hf_hub_ureq` | 2,308,944 B | +1,969,952 B | 65 | Smallest official-client probe; sync API. |
| direct `reqwest` rustls blocking | `reqwest_rustls_blocking` | 2,615,808 B | +2,276,816 B | 115 | Larger than `hf-hub` ureq and lacks Hub cache/revision semantics. |

Interpretation:

- Hand-rolled direct HTTP is not winning on binary size in this probe.
- `hf-hub` default features are not acceptable because they pull both client stacks and native TLS.
- `hf-hub` with `ureq` is the smallest measured official-client path and keeps the sync API.
- `hf-hub` with `tokio,rustls-tls` is still reasonable if the selected implementation needs async
  progress and already carries Tokio.

Corrected Loom TLS baseline:

- `crates/loom-cli` enables `tokio` for `serve` and depends on `rustls = 0.23` with
  `default-features = false` and `features = ["aws_lc_rs", "std"]`.
- `crates/loom-hosted` enables hosted TLS through `rustls` plus `tokio-rustls`, with the same
  `rustls` feature profile.
- baseline feature: `loom_tls_baseline`;
- baseline linked a small current-thread Tokio runtime, `rustls 0.23` with `aws_lc_rs,std`, and
  `tokio-rustls 0.26`;
- baseline size: 1,156,208 bytes;
- baseline normal deps: 16.

Measured on 2026-07-05:

| Probe | Feature set | Size | Delta over Loom TLS | Normal deps | Finding |
| --- | --- | ---: | ---: | ---: | --- |
| Loom TLS baseline | `loom_tls_baseline` | 1,156,208 B | 0 B | 16 | Tokio plus repo-matching rustls aws-lc control. |
| `hf-hub` defaults | `loom_tls_baseline,hf_hub_default` | 3,347,808 B | +2,191,600 B | 186 | Still broad because defaults include sync and async stacks plus native TLS. |
| `hf-hub` async rustls | `loom_tls_baseline,hf_hub_tokio_rustls` | 3,178,208 B | +2,022,000 B | 160 | Best async official-client profile measured here. |
| `hf-hub` blocking ureq | `loom_tls_baseline,hf_hub_ureq` | 3,124,464 B | +1,968,256 B | 73 | Smallest official-client profile over the repo TLS baseline. |
| direct `reqwest` rustls blocking | `loom_tls_baseline,reqwest_rustls_blocking` | 3,379,616 B | +2,223,408 B | 118 | Larger than both explicit `hf-hub` profiles in the corrected baseline. |

The previous Tokio-only baseline is useful only as a diagnostic control. DP1 uses the corrected Loom
TLS baseline because the CLI/hosted stack already standardizes on rustls with aws-lc.

Decision posture:

- Do not use `hf-hub` defaults.
- Prefer `hf-hub` over hand-rolled direct HTTP unless a later probe shows a materially smaller and
  safer direct client.
- The v1 enterprise implementation MUST use:
  `hf-hub = { version = "0.5", default-features = false, features = ["tokio", "rustls-tls"] }`.
- `hf-hub` with `ureq` is allowed only as a temporary CLI-only prototype shortcut. It is not the
  target architecture because downloads are daemon-visible async jobs with progress, cancellation,
  retries, status, and observability.

## 5. Cross-Platform Acquisition

`loom-inference` owns acquisition on every target.

Native:

- Use `hf-hub` with `default-features = false` and `features = ["tokio", "rustls-tls"]`, never
  defaults.
- Use the standard Hugging Face cache location by default.
- Support `HF_HOME` and the Hugging Face cache conventions rather than creating a separate model-byte
  cache.
- Do not persist cache inventory or completed download state. The shared cache is user-managed and may
  be mutated by tools outside Loom. Loom discovers local model availability from the current cache
  filesystem and Hugging Face metadata when it needs truth.

Browser / wasm:

- `hf-hub` is not used.
- Use platform fetch plus OPFS/IndexedDB or in-memory byte storage.
- Enforce a small-model cap for browser downloads.
- Reuse the same `loom-types` descriptors and job states.

Shared cache policy:

- The default cache is the shared Hugging Face cache.
- `loom inference model remove` is allowed to manage entries in `HF_HOME` or the standard Hugging
  Face cache when the requested paths resolve under the active Hugging Face cache root.
- `remove` MUST show what paths will be deleted in interactive mode and MUST support `--dry-run`.
- `remove` MUST refuse to delete paths outside the resolved Hugging Face cache roots.
- `remove` MUST be explicit that the cache is shared and user-managed. Loom does not know whether
  another client depends on a cache entry.
- Loom MUST NOT create application-owned state directories within the Hugging Face cache hierarchy.
  Durable instance and workspace-binding configuration belongs in the selected Loom workspace;
  installed model artifacts remain under the standard Hugging Face cache root.
- A deployment MAY configure a private cache root for sandboxed or managed environments.

Rationale: separate Loom-hosted model folders would duplicate multi-gigabyte files. The shared cache
is the expected user model-storage surface.

## 6.1 Instance Settings

Loom owns the public instance settings schema. Runtime crates are implementation details.

The settings model has three layers:

1. Portable settings shared by LLM and embedding providers where they make sense.
2. Runtime-specific settings for Ollama, llama.cpp, MLX, candle, hosted APIs, and future providers.
3. A structured escape hatch for provider-specific fields that Loom does not yet model.

Required preset behavior:

- Built-in presets include at least `fast`, `balanced`, `quality`, and `deterministic`.
- Presets are resolved into concrete settings before an instance is saved.
- Explicit `--set key=value` overrides are validated against the selected runtime and model kind.
- Hidden runtime defaults MUST NOT be the only record of behavior. The resolved settings are part of
  the instance record and are shown by CLI and JSON output.

## 7. Daemon and Standalone Behavior

The active download queue is host-cache-scoped, not `.loom`-store-scoped.

Current Loom daemons are store-scoped. 0062 adds a host inference coordinator role keyed by the
resolved Hugging Face cache root, not by a particular store. The coordinator owns only in-memory
active download state. It does not own the cache and does not persist a recovery queue.

Command behavior:

- If an inference coordinator is reachable, `loom inference model download` enqueues a job, prints the
  job id, and exits unless `--foreground` or a future `--follow` is supplied.
- If no coordinator is reachable, the CLI takes the cache lock and performs the download inline with
  progress on stderr.
- `loom inference model list` and `loom inference model status` read from the coordinator when it is
  reachable. Without a coordinator, only the inline CLI process can show live progress.
- `loom inference model cancel` requires a reachable coordinator for running jobs. There is no
  persisted queued job state to edit when the coordinator is absent.
- Only one process may actively download into a given cache root at a time. The cache lock is the
  Loom advisory boundary for Loom-managed operations only.

Ephemeral job states:

```text
queued
resolving
downloading
verifying
installed
failed
canceled
removing
removed
```

Progress reporting:

- Inline mode prints file-level progress and total job progress.
- Coordinator mode keeps active job progress in memory often enough that `list` can show live status
  while the coordinator is running.
- A future `--follow` option MAY stream coordinator progress to the terminal.

## 10. Runtime Support Matrix

Runtime meanings:

| Runtime | Role | Support posture |
| --- | --- | --- |
| `candle-safetensors` | Native Rust safetensors models and embedding encoders. | First-class for text embeddings; first-class for LLMs only for curated architectures that pass smoke. |
| `candle-gguf` | Native Rust GGUF loading through candle quantized support. | First-class for small/quantized LLMs when release smoke passes. |
| `mlx` | Apple MLX runtime and MLX-format weights. | Apple-only optional provider in `loom-inference`; never required by core or non-Apple builds. |
| `llama.cpp` | Native llama.cpp-compatible GGUF runtime. | Optional provider for broader GGUF compatibility when candle is insufficient. |
| `ollama` | External Ollama daemon over HTTP. | Remote/local service provider, not an in-process runtime. Loom integrates `ollama-rs` behind Loom-owned settings and instance records; Ollama owns its own model store. |
| `openai-compatible-http` | OpenAI-compatible hosted endpoint. | Remote provider path for LLMs and embeddings. |
| `hosted-api` | Multi-provider hosted API access. | `genai` adapter path for OpenAI, Anthropic, Gemini, Ollama-compatible, Bedrock, Vertex, OpenRouter, and similar API shapes. Not a local model artifact runtime. |

MLX support:

- MLX is a separate runtime profile, not a GGUF runtime and not a candle target.
- MLX support means an optional Apple-only provider that loads MLX-compatible weights through a
  refreshable native runtime bundle.
- The default Loom binary MUST run when the MLX bundle is absent. Non-Apple builds MUST NOT require
  MLX headers, libraries, or Apple frameworks.
- The raw MLX C bundle is staged outside git under `crates/loom-inference/native/mlx/<target>/` and
  contains `libmlx.dylib`, `libmlxc.dylib`, `mlx.metallib`, transitive local `*.dylib` runtime
  dependencies, and a manifest.
- A loadable Loom MLX runtime bundle also contains `libloom_mlx_adapter.dylib`. That adapter library
  is the Loom-owned ABI layer above upstream MLX C.
- A static-only MLX C prefix such as `/opt/mlx-c` with `libmlx.a`, `libmlxc.a`, and `mlx.metallib` is
  useful for compile-time experiments but is not the refreshable runtime bundle target. The runtime
  bundle target is shared libraries built from a matching MLX C source checkout with
  `BUILD_SHARED_LIBS=ON`.
- The real bundle builder runs on macOS with Xcode command line tools. A Linux container or VM build
  path is not a target for MLX because the runtime links Apple frameworks such as Metal, Foundation,
  QuartzCore, and Accelerate.
- The builder script owns the MLX C source lifecycle for local development: clone when missing,
  fetch and check out the requested ref when present, and provide a clean rebuild mode for the
  managed build/install directories.
- The manual MLX smoke compiles a checked-in C probe against the cloned MLX C headers and staged
  dynamic bundle. It runs a minimal array operation when MLX can access a Metal device. It returns
  unavailable with exit code 77 when MLX loads but the host session cannot expose a Metal device.
- The preferred runtime contract is a Loom-owned adapter ABI over the staged MLX C bundle. `mlx-rs`
  remains useful evidence for API shape and prototype behavior, but its current published build
  vendors and statically links MLX C, which does not satisfy the refreshable external-library target.
- The Loom-owned MLX adapter ABI starts at version 1 and requires these exported symbols:
  `loom_mlx_adapter_abi_version`, `loom_mlx_adapter_runtime_info`, `loom_mlx_adapter_load_llm`,
  `loom_mlx_adapter_load_text_embedding`, `loom_mlx_adapter_release_handle`, and
  `loom_mlx_adapter_last_error`.
- `loom_mlx_adapter_runtime_info` returns a non-null `const char *` that points to static
  NUL-terminated UTF-8 JSON owned by the adapter. The JSON MUST include `schema_version`,
  `adapter_abi_version`, and `runtime`; Loom parses it once at adapter load time, validates
  `schema_version = 1` and `adapter_abi_version = 1`, and caches the typed result.
- The current source-backed MLX C surface is a tensor and array runtime with safetensors and GGUF
  loading primitives. It does not provide a tokenizer, BERT, sentence-transformer, Qwen, or general
  transformer text-generation runtime. Queue 6 MUST NOT hand-roll those model graphs inside
  `loom-inference`; MLX provider execution requires a Loom MLX adapter library that implements a
  curated model family or a higher-level MLX model runtime with explicit settings.
- `loom inference model list --remote` MUST not imply a GGUF model can run under MLX.
- `loom doctor` reports MLX as `ok`, `not_compiled`, `unsupported_os`,
  `static_only_runtime`, `missing_adapter_library`, or `missing_runtime`.

Apple built-in inference support:

- Core ML is the Apple built-in inference framework path. It is distinct from MLX and distinct from
  candle Metal.
- MLX is Apple-built and Apple Silicon-focused, but it is an external runtime and model ecosystem,
  not the operating-system inference framework.
- Candle Metal uses Apple's Metal GPU compute stack through candle. It does not load MLX models and
  does not provide Core ML model scheduling.
- Core ML is represented as `RuntimeKind::CoreMl` / `core-ml`. It is optional, never a default
  requirement for non-Apple builds, and requires macOS for CLI doctor and runtime availability.
- Core ML model artifacts are accepted as `.mlmodel`, `.mlpackage`, or files inside a compiled
  `.mlmodelc/` bundle. The provider bridge is separate target work and MUST keep the default Loom
  binary portable on non-Apple targets. Non-Apple builds MUST NOT require Core ML headers, Apple
  frameworks, or Apple toolchains.

Ollama / llama.cpp support:

- Ollama is supported as an external service provider through `ollama-rs`, wrapped by Loom-owned
  instance and settings types.
- `loom inference model download` does not move Ollama-managed model blobs into the Hugging Face
  cache.
- `loom inference model list --local --runtime ollama` MAY call Ollama's local model list endpoint.
- `loom inference model download --runtime ollama <name>` MAY map to Ollama pull in a later
  implementation, but it is not the same storage path as Hugging Face downloads.
- Ollama settings MUST expose the typed Ollama controls available through `ollama-rs`, including
  `mirostat`, `mirostat_eta`, `mirostat_tau`, `num_ctx`, `num_gpu`, `num_thread`, `repeat_last_n`,
  `repeat_penalty`, `temperature`, `seed`, `stop`, `tfs_z`, `num_predict`, `top_k`, `top_p`,
  `min_p`, `keep_alive`, and `think`, plus a validated provider-specific extra map.
- llama.cpp is the selected local GGUF execution path for official GGUF artifacts that rely on GGUF
  tokenizer metadata or llama.cpp model-family support.
- llama.cpp support is a Loom-managed optional external dynamic runtime bundle loaded through the
  `loom-native` boundary, not a default linked dependency of the main Loom binary.
- The raw upstream llama.cpp bundle contains shared runtime libraries such as `libllama`,
  `libggml`, `libggml-base`, and `libggml-cpu`. A loadable Loom bundle also contains a Loom-owned
  adapter library named `libloom_llama_cpp_adapter.dylib`, `libloom_llama_cpp_adapter.so`, or
  `loom_llama_cpp_adapter.dll`.
- The Loom llama.cpp adapter ABI starts at version 1 and requires these exported symbols:
  `loom_llama_cpp_adapter_abi_version`, `loom_llama_cpp_adapter_runtime_info`,
  `loom_llama_cpp_adapter_load_llm`, `loom_llama_cpp_adapter_complete`,
  `loom_llama_cpp_adapter_release_handle`, and `loom_llama_cpp_adapter_last_error`.
- `loom_llama_cpp_adapter_runtime_info` returns a non-null `const char *` that points to static
  NUL-terminated UTF-8 JSON owned by the adapter. The JSON MUST include `schema_version`,
  `adapter_abi_version`, and `runtime`; Loom parses it once at adapter load time, validates
  `schema_version = 1` and `adapter_abi_version = 1`, and caches the typed result.
- LLM load and completion calls use Loom-owned JSON request and response schemas. Request JSON is
  passed as caller-owned NUL-terminated UTF-8. Completion response JSON is written into a
  caller-owned buffer with a required-length out parameter so the ABI does not transfer allocation
  ownership across the dynamic-library boundary.
- llama.cpp adapter status codes start with `0 = ok`, `1 = invalid-request`,
  `2 = buffer-too-small`, and `3 = runtime-error`. Non-zero statuses are mapped through
  `loom_llama_cpp_adapter_last_error`.
- The default Loom binary MUST run when the llama.cpp bundle is absent. Non-GGUF users MUST NOT need
  llama.cpp headers, libraries, clang, bindgen, CMake, or native llama.cpp toolchains.
- The local development bundle script owns the managed llama.cpp source lifecycle: clone when
  missing, fetch and check out the requested ref when present, and provide a clean rebuild mode for
  the managed build/install directories.
- `llama-cpp-2` remains useful API and build-system evidence, but Loom's public runtime contract is
  the Loom-owned adapter ABI and the managed external bundle, not direct default linkage to
  `llama-cpp-2`.

Hosted API support:

- Hosted API access is backed by `genai` and is separate from local downloaded model artifacts.
- `genai` is used for broad provider protocol coverage and normalized chat/embedding calls.
- `genai` does not replace `ollama-rs` for Ollama-specific lifecycle and full Ollama settings.
- The Loom instance schema remains the stable public contract. `genai` model names, adapter kinds,
  provider-specific extras, and authentication details are stored as Loom settings, not exposed as
  public crate types.

## 11. Facet Integration

Current source-backed data-facet consumer:

- `vector` is the direct consumer. Source-aware vector upsert uses the text embedding provider, stores
  source text, and records the embedding model profile.
- `loom vector text upsert` and `loom vector text query` resolve configured text-embedding instances
  through `loom-inference`, activate the installed provider, and project source-text hits through the
  vector facet.

Target integrations:

- Daemon-coordinated vector text workflows expose foreground follow and durable background status once
  the download coordinator path is complete.
- `vector_embedding_model` continues to report the vector-facing model profile, not the full download
  descriptor.
- GraphRAG uses downloaded text embeddings through the vector facet. It does not get a separate
  storage path for embeddings.
- Search may later add semantic or hybrid query expansion through a text embedding provider, but no
  direct search-facet dependency is source-backed today.
- LLM consumers use `Llm`. Completion output is derived operational data unless a higher-level facet
  stores prompt, model id, and output explicitly.

Provider resolution:

```text
configured model ref
  -> local descriptor
  -> doctor/runtime compatibility check
  -> provider instance
  -> Llm or TextEmbedding handle
```

No data facet should import `hf-hub` or know where files are cached.

Non-text source integration:

- 0068 owns PDF, image, audio, video, and multimodal source extraction, source unit identity,
  provenance, and modality metadata.
- 0062 MAY add acquisition and activation support for curated multimodal models only after those
  models declare compatibility with a 0068 provider contract.
- Until 0068 source units and provider contracts are source-backed, 0062 remains limited to `llm` and
  `text-embedding` model kinds.

## 12. Security, Policy, and Licensing

Download policy:

- Never download a model implicitly during a data read.
- Require an explicit user action for network downloads.
- Surface license, gated status, model size, runtime profile, target cache path, and delete policy
  before download when the command is interactive.
- Support non-interactive approval flags for CI and managed deployments.
- Use Hub tokens from key sources or process environment only for request auth; never store them.
- Do not execute remote model code. The first pass supports data files only: safetensors, GGUF,
  tokenizer JSON, config JSON, and allowlisted metadata.
- Prefer safetensors for tensor weights. Pickle and arbitrary Python model files are out of scope.

Removal policy:

- `remove` deletes entries from the shared Hugging Face cache only when they are under the resolved
  cache roots and the user requested deletion.
- `remove --dry-run` prints every file and metadata record that would be removed.
- Removing a model is a user-directed shared-cache action. Loom does not infer cross-tool ownership
  from local configuration and does not retain a private model-reference inventory.
- Shared-cache deletion is allowed because the user asked for shared cache semantics. The CLI must make
  the deletion explicit rather than pretending model files are Loom-private.

Operational guidance:

- Managed deployments SHOULD set `HF_HOME` to an explicit host or container volume so model bytes
  survive process restarts. Download status is process memory and is not a recovery log.
- Managed deployments SHOULD pre-download and verify approved model revisions during image build,
  host bootstrap, or a controlled maintenance window. Ordinary read paths MUST NOT trigger network
  model downloads.
- Hub tokens are request credentials, not model inventory. They MAY come from the process environment,
  platform credential stores, or future Loom key sources, but Loom MUST NOT write Hub tokens into the
  cache, download status, or model instance settings.
- Gated models require authenticated access and a known allowed gated status from a source-backed
  catalog or current Hub metadata. A model with unknown gated status MUST NOT be reported as runnable
  by doctor.
- License status is part of compatibility. A model with an unknown or blocked license MUST NOT be
  reported as runnable by doctor, even when the files are present and the runtime is compiled.
- Operators SHOULD use `loom inference model remove --dry-run` before deletion in shared-cache
  deployments. Non-interactive deletion MUST show the cache root and removed paths in command output
  or logs.
- Instance settings SHOULD be treated as deployment configuration. A managed deployment should pin the
  instance name, model revision, runtime, preset, and explicit setting overrides rather than relying on
  changing defaults.

Optional native runtime bundle policy:

- Optional native runtime bundles are separate release artifacts from the default `loom` executable.
  The default `loom` binary MUST start and run without these bundles.
- Local inference execution backends are opt-in CLI build features named `backend-candle-cpu`,
  `backend-candle-cuda`, `backend-core-ml`, `backend-llama-cpp`, and `backend-mlx`. The default CLI
  feature set MAY include model acquisition and provider-control features such as `hf-hub`,
  `genai`, and `ollama-rs`, but MUST NOT require a local tensor execution backend. This preserves a
  portable default binary while still allowing a single binary to be built with selected execution
  backends.
- MLX bundles are Apple-only artifacts. Release automation MUST NOT attach an MLX bundle to Linux,
  Windows, wasm, or non-Apple target packages.
- MLX bundles MUST be built from a pinned upstream `mlx-c` Git revision using
  `scripts/inference-mlx-runtime.sh clean-build`, and release materials MUST record the upstream
  revision, target triple, file list, file sizes, linkage manifest, and SHA-256 checksums.
- MLX bundles MUST include every local runtime `*.dylib` required by `libmlxc.dylib` and
  `libmlx.dylib`, the matching `mlx.metallib`, a manifest, and the Loom adapter library once the
  adapter is implemented.
- Generated MLX bundles MUST remain outside git. They MAY be attached to release assets, distributed
  by an installer, or installed by an enterprise deployment manager into a configured bundle directory.
- `loom doctor` is the validation surface for runtime deployment. It MUST report missing
  bundles, static-only prefixes, missing runtime files, missing adapter library, unsupported host, and
  loadable states.
- Refresh cadence follows upstream security and compatibility needs. A bundle refresh MUST be treated
  like a native dependency update: rebuild, run the manual smoke on a host with Metal access, update
  checksums, update the manifest, and publish release notes for the upstream revision change.
- FIPS release materials MUST list optional native runtime bundles separately from the FIPS binary.
  An optional MLX bundle MUST NOT expand any FIPS compliance claim unless a separate compliance review
  explicitly includes that native bundle and its Apple framework dependencies.

## 14. Decision Points

### DP1 - Native Hugging Face downloader

**Question:** Should Loom use the official Rust `hf-hub` crate or hand-roll direct HTTP downloads?

**Context:** Downloads need revision resolution, cache reuse, progress reporting, metadata, and future
Hub auth. The size probes show direct `reqwest` rustls blocking is larger than the smallest `hf-hub`
probe, while lacking Hub cache/revision semantics. `hf-hub` defaults are too broad.

**Examples:** A text embedding download needs `config.json`, `tokenizer.json`, and `model.safetensors`
from one pinned revision. A GGUF LLM download may need one large `.gguf` file plus tokenizer metadata.

**Options:**

- Use `hf-hub` natively with `default-features = false` and `features = ["tokio", "rustls-tls"]`.
- Use `hf-hub` natively with `default-features = false` and `features = ["ureq"]`.
- Use direct `reqwest` GETs for every native download.
- Use both as equal code paths.

**Decision:** Use `hf-hub` natively with `default-features = false` and
`features = ["tokio", "rustls-tls"]`.

**Recommendation:** Treat the async `hf-hub` profile as the v1 enterprise target. The small binary
delta in favor of `ureq` does not outweigh daemon-owned progress, cancellation, retries, status, and
observability. Keep direct GET only as the browser/fallback path.

**Consequence of deferring:** Reopening this decision would block the downloader shape and risks two
cache/progress paths, one blocking and one daemon-owned async.

### DP2 - Cache ownership

**Question:** Should downloads go into Loom's cache or the user's shared Hugging Face cache?

**Context:** Shared cache reuse avoids duplicate multi-gigabyte model files. The risk is deletion:
`loom inference model remove` may delete files another tool expects.

**Examples:** A user may already have `HF_HOME` populated by Python, candle examples, or another local
model tool. Duplicating those files for each Loom-hosted folder wastes disk.

**Options:**

- Loom-owned cache by default, scan shared Hugging Face cache read-only.
- Shared Hugging Face cache by default.
- Always duplicate files into both caches.

**Decision:** Shared Hugging Face cache by default.

**Recommendation:** Implement `remove --dry-run`, explicit path display, cache-root guards, and active
provider guards so deletion is visible and bounded.

**Consequence of deferring:** The implementation either duplicates large model files or ships unsafe
delete behavior without an explicit owner decision.

### DP3 - LLM first-class scope

**Question:** Should the download manager support only text embeddings first, or both LLM and
text-embedding models from the start?

**Context:** Existing source already has completion and embedding seams. 0050 makes embeddings
first-class. 0051 left local LLM scope unsettled, but the owner decision for 0062 is to support both
model kinds fully.

**Examples:** `sentence-transformers/all-MiniLM-L6-v2` should immediately feed the vector facet.
`Qwen/Qwen2.5-0.5B-Instruct-GGUF` should be downloadable, diagnosable, and runnable through at least
one local LLM runtime when the catalog marks it supported.

**Options:**

- Text embeddings only in 0062.
- Both model kinds in catalog/download/CLI, but only text embeddings are provider-backed in the first
  runtime slice.
- Both model kinds fully provider-backed in the first release slice.

**Decision:** Both model kinds, fully backed.

**Recommendation:** Fully backed means at least one release-smoked local LLM runtime for the curated
small-model tier, not every Hugging Face LLM.

**Consequence of deferring:** CLI terminology and catalog records will churn, and users will see
downloadable LLMs that cannot be run through Loom.

### DP4 - Apple built-in inference path

**Question:** Should Core ML be added to 0062, and if so when should it start?

**Context:** Core ML is the Apple built-in inference framework path. MLX is a separate Apple-built
runtime and candle Metal is a Rust runtime over Metal GPU compute. Treating those as the same backend
would hide different model formats, settings, packaging, and hardware-routing behavior.

**Examples:** A Core ML provider would run Core ML packages and rely on Apple's platform scheduler. An
MLX provider runs MLX-compatible weights through the staged MLX C bundle. A candle provider runs
curated safetensors or GGUF models through Rust candle runtimes.

**Options:**

- Add Core ML as a first-class Apple runtime in this queue after MLX dynamic loading is source-backed.
- Exclude Core ML and focus Apple support only on candle Metal plus optional MLX.
- Defer Core ML to a separate future queue.

**Decision:** Add Core ML as first-class Apple built-in inference work in Queue 6 after MLX dynamic
loading is source-backed.

**Recommendation:** Keep Core ML separate from MLX and candle Metal, but keep it in this queue once
the native MLX loading boundary is real. Core ML should have its own model format policy, conversion
rules, settings, packaging, and doctor checks.

**Consequence of deferring:** Loom would support Apple hardware through candle and optional MLX only,
and would continue to lack Apple operating-system inference or Neural Engine eligibility through Core
ML.
