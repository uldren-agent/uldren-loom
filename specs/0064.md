# 0064 - Unified Search / Discovery

**Status:** Draft, target. **Version:** 0.1.0.
**Capability:** `unified-search`.

**Depends on:** 0033 (search facet / FTS + native Tantivy engine), 0017 (vector layer - exact-is-contract,
accelerators rebuildable), 0040 (GraphRAG composition), 0016 (graph engine), 0023 (columnar), 0061 §10
(cross-facet search interface, engine ladder §10.2, decoupling rule §10.3), 0050 (embedding providers),
0051 (LLM providers), 0062 (inference model downloads - `TextEmbedding` acquisition), 0026-0028
(principals + access control + PEP), 0035 (durable delivery / cursors), 0036 (locking + embedded
artifact single-writer). **Relates to:** _QUEUE2 (Tier-2 tasks 280b + vector compatibility ports),
SLACKISH §2.1 / 0061 §3 (envelope-field visibility rule).

This spec defines **unified search and discovery** over Loom: the engine ladder behind the
`substrate_search` interface, the ranking/fusion contract, the embedding pipeline, the whole-Loom
`loom search` scope, the decoupling contract Loom Studio depends on, and the shared-index contract for
the Tier-2 edge ports.

It is **not a new storage facet.** Like 0040 (GraphRAG), it is a *composition*: it composes the search
facet (0033), the vector layer (0017), and GraphRAG (0040) behind one interface, over the versioned
engine and the 0061 operation substrate. It defines no new storage class. `substrate_search` and
`loom search` are the same interface (0061 §10.3); this spec is that interface's owning design - the one
home for the ladder, the fusion contract, the embedding pipeline, and the edge-port shared-index rule.

Implementation status: target design with source-backed lower rungs. Scan-backed `substrate_search`
v1 is source-backed by 0061/0043. The optional `loom-tantivy` native lexical rung and search-specific
derived-artifact lifecycle are source-backed by 0033 and current source. The reusable vector
accelerator foundation for the semantic rung is source-backed by 0017 and current source. The reusable
embedding projection job/freshness model is source-backed in `loom-substrate::search`, and the durable
local embedding rebuild/status/no-keys lifecycle is source-backed in `loom-store`. Incremental
diff-scoped re-embedding planning over the `LMDIFF` envelope is source-backed in
`loom-substrate::search`. `loom-mcp` source-backs a bounded semantic rung for `substrate_search` over
an existing 0017 vector projection when the caller supplies a query vector and matching embedding
model identity. Text-to-vector query embedding, ANN/HNSW semantic execution, MCP/whole-Loom hybrid
execution over multiple live rungs, graph fusion, whole-Loom discovery, and edge-port parity remain
target work.
The deterministic RRF fusion primitive, `match_via` provenance merge, and tie-break chain are
source-backed in `loom-substrate::search`; callers must pass already PEP-filtered rung lists.
Decision Points DP1-DP8 are decided in this document.

## 2. Shape and Scope Model

`unified-search` is a facade + query planner + projection/embedding maintenance layer, not storage.

Two names, one interface, one Loom:

- **`substrate_search`** - the in-substrate cross-facet query interface Studio profiles and the
  assistant code to (0061 §10). Default scope is the calling store/organization.
- **`loom search`** - the CLI/tool projection of the *same interface* at **whole-Loom** scope: it spans
  everything in the one Loom (files, documents, mail, calendar, metadata, text indexes, semantic
  indexes) rather than a single organization. This is **not** cross-Loom federation across multiple Looms;
  there is one Loom, and `loom search` is its whole-store projection.

Both take an **optional `scope`** that *narrows* breadth. Scope is a within-Loom selector, not a
federation key:

```text
scope (optional; narrows from whole-Loom default):
  workspace   restrict to one workspace
  collection  restrict to one collection within a workspace
  facet       restrict to one facet class (document | columnar | graph | ledger | mail | calendar | ...)
  (absent)    whole Loom
```

`loom fts` stays the separate, narrower surface: collection-scoped, lexical-only, for precision and
speed (0033 decision 4, _QUEUE2 batch 280). `loom search` is the discovery verb - broad by default,
laddered, fused. Choosing `loom fts` vs `loom search` is choosing precision-in-a-collection vs
discovery-across-the-Loom, not choosing an engine (the engine is selected behind the interface, §5).

The `as_of`/`history` versioning scope (0061 §10.1) is orthogonal to the breadth `scope` above and
composes with it: a query carries both a breadth selector (workspace/collection/facet) and a version
selector (latest | as_of | history).

## 4. Ranking and Fusion Contract

### 4.1 Reciprocal Rank Fusion (RRF), rank-based only

**DP2 - decided: RRF-only, rank-based; no cross-engine score normalization.**

Hybrid fusion combines the per-rung ranked lists using **reciprocal rank fusion**: each result's fused
score is the sum over contributing rungs of `1 / (k + rank_in_that_rung)`, where `rank_in_that_rung` is
1-based and `k` is a damping constant.

```text
rrf_score(hit) = Σ_over_rungs  1 / (k + rank_r(hit))
k default = 60 ; overridable per query (rrf_k), never per deployment silently
```

Rationale: a lexical BM25 score (unbounded, corpus- and analyzer-dependent) and a vector cosine/metric
score (bounded, metric-dependent) are **not comparable magnitudes**. Normalizing and adding them
(min-max, z-score) is fragile, corpus-sensitive, and non-deterministic across shards and across `as_of`
roots. RRF depends only on *rank position within each rung*, which is well-defined regardless of engine,
shard layout, or root. This is the enterprise-performant-DRY choice: **one** fusion algorithm, no
per-shard/per-root score-calibration pass, deterministic and auditable.

- Normalized / weighted-score fusion is **not** a contract mode. It is explicitly rejected to keep a
  single deterministic fusion path (no second calibration code path to maintain or conform).
- **Raw per-rung scores are still returned** on each hit for *display and debugging* (`raw_score`,
  `rung`), but they are never inputs to fusion.
- Per-rung rank truncation (each rung contributes its top-N before fusion) is a performance bound, not a
  ranking knob; `N` (`rung_depth`, default sized to `limit + offset` with a fixed multiplier) is
  declared in the response so a caller can see the recall bound.

### 4.2 `match_via` provenance

Every hit carries a `match_via` provenance tag so a caller knows *why* it matched without knowing which
engine ran (0061 §10.2):

```text
match_via ∈ { lexical | semantic | hybrid | graph }
```

- A hit surfaced by a single rung carries that rung's tag.
- A hit surfaced by more than one rung (fused) carries `hybrid`, plus a `contributing_rungs` list
  (e.g. `[lexical, semantic]`) so provenance is not lossy.
- A hit pulled in by graph expansion (0040) carries `graph`, plus the `via_edge`/`via_entity` that
  reached it, so RAG context is explainable and auditable (0040's provenance goal).

### 4.3 Deterministic tie-breaks

**DP3 - decided: tie-break chain is `match_via` priority -> `entity_id` -> `as_of` root.**

When two hits have equal fused RRF scores, order is broken deterministically and engine-independently:

```text
1. match_via priority:  lexical > semantic > graph   (hybrid ranks at its best contributing rung)
2. entity_id:           stable ascending byte order
3. as_of root:          the root the hit was computed against (stable)
```

Raw engine scores are **not** a tie-break input (that would reintroduce the cross-engine comparability
problem RRF avoids). Recency is **not** a tie-break input (it couples ranking to mutable timestamps).
The chain is total and reproducible: the same corpus at the same root returns the same order on every
engine and every shard layout.

### 4.4 `as_of` / `history` interaction per rung

The versioning scope (0061 §10.1) composes with each rung. Every hit reports the root it was computed
against, so staleness is badged, never hidden.

| Rung | `latest` | `as_of(root)` | `history` |
| --- | --- | --- | --- |
| scan | scan current root | scan the revision set reachable from that root | scan all retained revisions/operations |
| lexical (Tantivy) | query the index projection of the current root; `INDEX_NOT_READY` if stale | retained index snapshot for that root, else scan fallback (same contract, 0061 §10.1) | only where the collection's index policy retains prior body revisions (0061 §10.1); append-only content is inherently historical |
| semantic (vector) | ANN over the embedding projection of the current root; `INDEX_NOT_READY` until the embed worker catches up | embeddings pinned to that root where retained, else exact scan over re-embedded reachable set | same body-revision-retention rule as lexical; append-only content already historical |
| hybrid | RRF over whichever of lexical/semantic are available at that root; degrades if one is absent | same, at the historical root | same |
| graph (0040) | traversal over the KG at the current commit | traversal pinned to that commit (0040's reproducible retrieval) | bitemporal edges (valid-time vs commit-time) where the KG models them |

Rule of thumb (encoded in tool descriptions, 0061 §10.1): "find X" -> latest; "where did we discuss X"
-> history (append-only content answers in latest by design); "what did it say at the release cut" ->
as_of. A rung that cannot honor the requested version scope for a given collection reports it as a
degraded result (§8.3), never a silent empty or a silent latest-substitution.

## 5. Engine Ladder Selection

The ladder is pinned in 0061 §10.2; this section states the selection contract. Each rung is a
rebuildable accelerator behind the same interface (the 0017 philosophy generalized: exact/scan is the
portable contract, every faster rung is optional derived state).

```text
scan  ->  lexical (Tantivy)  ->  semantic (vector ANN)  ->  hybrid (RRF)  ->  graph-expanded (GraphRAG)
```

Selection rules:

- **Mode requested by the caller** (`text | regex | semantic | hybrid | similar_to`) sets the *ceiling*
  of the ladder. `hybrid` is the default when both lexical and semantic indexes exist for the scope.
- **Availability sets the floor.** A rung is available only if its index exists, is ready
  (not `INDEX_NOT_READY`), and the caller holds the keys the rung needs (§6.4). When a requested rung is
  unavailable, the engine falls **down** the ladder and marks the result degraded (§8.3) - never a
  silent empty result, never a silent pretend-match.
- **No size-threshold ranking switch.** Following 0033 N.7, the engine does not silently change ranking
  semantics at a corpus-size boundary. Lexical is native-authoritative when present, portable-reduced
  (scan) otherwise, surfaced by `reduced` (0033 N.2). Selecting the vector accelerator vs exact vector
  scan is the 0017 `AcceleratorPolicy` choice and is reported, not hidden.
- **`similar_to(entity_id)`** is the find-related surface: it runs the semantic rung seeded by the
  entity's own embedding, optionally graph-expanded, with no free-text query.

## 6. Embedding Pipeline

### 6.1 Content selection

**DP4 - decided: embed versioned bodies + append-only content by default; per-collection opt-out.**

Default embedded content:

- **Versioned bodies** - pages, spec bodies, descriptions (latest revision by default; prior revisions
  only where the collection's `history` index policy retains them, per 0061 §10.1).
- **Append-only content** - annotations, operations, messages, ledger-backed discussion. This is what
  makes "where did we discuss backpressure?" answerable semantically (0061 §10.1's primary worked
  example), because append-only content's latest index *is* its full history.

Per-collection **opt-out** (`embed_policy = off`) turns semantic indexing off for a collection whose
content is not worth the index cost (e.g. high-churn machine logs). Body-revision embedding beyond
latest is per-collection opt-in for the same size/retention reasons the lexical `history` policy carries
(0061 §10.1). Structured non-text (raw columnar numerics, kv blobs) is not embedded; it is reachable via
lexical/scan and predicates.

### 6.2 When embeddings are generated

**DP5 - decided: async keyed index-worker, enqueue-on-write; reuse the 0033/0017/0040 accelerator
lifecycle. No new pipeline.**

The embedding projection is a **derived, rebuildable, identity-excluded** accelerator with exactly the
lifecycle 0033 N.6/N.8 and 0017 §5 already define - this spec forks no second mechanism (DRY):

- **Enqueue on write.** A write to embedded content enqueues an embed job keyed to the content digest.
  The write path does not block on embedding (writes stay fast; write latency is not coupled to provider
  latency; blind hosts that cannot embed are never blocked, §6.4).
- **Async under the daemon.** The daemon runs the embed/index build asynchronously and returns a
  status/handle (mirroring 0033 N.8). A stateless invocation runs it synchronously then unloads.
- **Coalescing + snapshot-stamped.** Builds are idempotent and coalescing (0033 N.8): a build in flight
  attaches rather than starting a second. Each build stamps the content-digest snapshot and the
  embedding model identity + engine version it built from.
- **Incremental via the commit diff.** On new commits, only the nodes/entities the 0003b/0040 diff
  reports changed are re-embedded (0040 RD4) - not a full rebuild.
- **Rebuild, never trust blindly.** The projection is copy-carried but sync-rebuilt: `push`/`pull`/
  `clone` move the DAG only; the receiver rebuilds (0017 §5, 0033 N.6). It is never versioned or synced
  and never part of canonical identity (the determinism spine: ANN/GPU output is local-only, 0040 §4,
  0050).

### 6.3 Freshness / staleness signalling

Staleness is always observable; the engine never lies about currency.

- A query needing the semantic rung against a projection that is missing, mid-build, or stamped to an
  older content digest / model identity than the current root returns **`INDEX_NOT_READY`** for that
  rung (0033 N.8) - surfaced as a degraded hybrid result (§8.3), not a silent drop to lexical without
  notice.
- Every hit reports the root and the model identity it was computed against; a model-identity or
  engine-version shift marks the projection stale and triggers rebuild before trust (0033 N.6, 0050
  per-workspace model identity).
- `substrate_search` / `loom search` responses carry an `index_status` block per rung
  (`ready | building | stale | not_built | no_keys`) so a caller (Studio, `loom search`) can badge
  freshness rather than present stale results as current.

Current source backs the reusable embedding projection job and freshness model in
`loom-substrate::search`: projection keys, content/model/engine stamps, queued/building/ready/stale/
failed/no-keys states, coalescing identity, and freshness comparison. Current source also backs the
durable-local rebuild lifecycle in `loom-store`: embedding-specific artifact keys, content/model/engine
stamps, begin/finish/fail rebuild calls, coalescing in-flight rebuilds, stale status, ready status, and
the blind-host no-keys state. `loom-substrate::search` also backs the incremental planner that consumes
the canonical `LMDIFF` envelope and emits unit rebuild, unit removal, or coarse collection rebuild
actions for embed-eligible text-bearing facets. Semantic vector build execution remains target work.
The source-backed MCP `substrate_search` semantic path runs exact 0017 vector search only when an
existing vector projection has a recorded embedding model profile and the request supplies a matching
`query_model_id` and `query_weights_digest`; otherwise it returns a degraded result with
`index_status.semantic` set to `not_built` or `stale`.

### 6.4 Provider posture and the keyed-only constraint

- **Provider via 0050 (DP6 - decided).** Embeddings are produced through the 0050 `TextEmbedding` seam;
  the model is resolved via 0062 (downloaded local model) or an OpenAI-compatible HTTP provider. Default
  posture is **local in-process embedding** (candle / MLX on the `ios` path) for the curated small
  models; hosted OpenAI-compatible embedding is an opt-in provider, never the silent default. Model
  identity is per-workspace (0050); mixing models within one vector set is a rebuild, not a merge.
- **Keyed-only (reconciled with the envelope-visibility rule, 0061 §3 / SLACKISH §2.1).** Embedding
  requires plaintext, and lexical tokenization requires plaintext. A **blind host cannot embed or
  tokenize** - it has envelope metadata (`operation_kind`, ids, digests, sizes, policy labels) but never
  payload. Therefore:

  ```text
  blind host          may:  store opaque vectors, store the lexical index blob (encrypted, opaque),
                            sync them, route/rate-limit by envelope metadata, run the scan rung over
                            envelope metadata only (degraded-blind)
  keyed worker /      must: produce embeddings, build the Tantivy lexical index, extract/preview text,
  key-holding client        run semantic/lexical/hybrid/graph rungs over plaintext
  ```

  This is exactly the 0061 §3 feature ledger: "search indexing, previews, text extraction … require a
  keyed worker or key-holding client." A keyless deployment therefore falls to the scan rung over
  visible envelope metadata and is marked degraded (§8.3). The recommended deployment (SLACKISH) keeps
  keys and embedding local or in a tenant-controlled keyed worker while a blind Loom Cloud stores the
  opaque vectors and lexical blobs and relays sync.

## 7. Permission-Scoped Fusion

**DP7 - decided: pre-filter per rung, then fuse. No post-filter stage.**

A whole-Loom `loom search` spans workspaces/collections/facets over which the caller's ACL grants
(0026-0028 + PEP) vary. Permission filtering happens **before** fusion:

- The caller's authorization predicate (from the PEP, expressed in the 0061 §14 predicate tree) is
  **pushed into each rung's own query**: the 0017 metadata pre-filter, the 0033 `bool`/`term` filter,
  the graph edge/label filter. Only authorized hits ever enter RRF.
- **No existence leak.** Because forbidden hits never enter the ranked lists, they cannot leak through
  rank position, result counts, or RRF score. Post-filtering after fusion is rejected precisely because
  it leaks existence via counts/ranking and wastes ranking work on content the caller cannot see.
- **DRY + performant.** Pre-filter reuses each rung's *existing* predicate-filter surface rather than
  adding a redundant post-fusion filter stage, and it does zero ranking work on forbidden content.
- Current source backs the fusion primitive in `loom-substrate::search`: it accepts already-filtered
  rung lists, computes RRF from rank positions, merges duplicate hits as `hybrid`, preserves
  contributing rung provenance, and applies deterministic tie-breaks.
- **Graph expansion honors the predicate.** 0040 graph traversal applies the same authorization
  predicate as an edge/label guard during expansion, so graph-pulled context cannot surface entities the
  caller lacks grants on. (This closes the graph-expansion leak surface rather than deferring it.)
- The PEP remains authoritative and can enforce blind at the sequencer (0061 §3); pre-filter is defense
  applied at query planning, not a replacement for the PEP.

## 8. Decoupling Contract (what Studio depends on)

Loom Studio and every Surface code to this contract and **never** learn which engine answered a query or
whether the scope was one organization or the whole Loom (0061 §10.3). Swapping or upgrading any rung is
invisible to every caller.

### 8.1 Request

```text
search request:
  query:      string                       # free text; ignored for similar_to
  query_vector: optional List<f32>         # source-backed MCP path for semantic mode when already embedded
  query_model_id: optional string          # required with query_vector for semantic readiness
  query_weights_digest: optional string    # compared with the vector projection model profile
  mode:       text | regex | semantic | hybrid | similar_to     # default hybrid
  similar_to: optional entity_id           # required iff mode = similar_to
  scope:      optional { workspace? collection? facet? }        # breadth; absent = whole Loom
  version:    latest | as_of(root|seq) | history               # default latest (0061 §10.1)
  filter:     optional predicate tree (0061 §14)               # caller filter; ACL predicate added by PEP
  limit:      u32
  offset:     u32
  rrf_k:      optional u32                 # default 60
  rung_depth: optional u32                 # per-rung truncation before fusion
  facets / highlight / aggregations: as 0033 QueryRequest
```

### 8.2 Response

```text
search response:
  hits: [
    {
      facet, collection, entity_id, field,
      snippet, offsets,
      scope_context: { owning_entity, status_fields, refs },   # 0061 §10 typed hit
      root,                              # the root this hit was computed against (staleness badge)
      match_via,                         # lexical | semantic | hybrid | graph  (§4.2)
      contributing_rungs,               # e.g. [lexical, semantic] when fused
      via_edge / via_entity,            # present when match_via = graph
      fused_score,                       # RRF score (§4.1)
      raw_score, rung                    # per-rung raw score, display/debug only (never fused)
    }
  ]
  engine: { rungs_available, rung_selected_ceiling, rrf_k, rung_depth }
  index_status: { lexical, semantic, graph } each ∈ ready|building|stale|not_built|no_keys
  reduced:  bool          # 0033 N.2 portable/degraded lexical path
  degraded: { is_degraded, reason }     # §8.3
```

### 8.3 Degraded-mode signalling

Degraded means: the engine could not run the requested ceiling of the ladder and fell down it. It is
**always signalled, never silent, never empty-when-it-should-not-be**:

```text
degraded.reason ∈ {
  no_keys,           # blind/keyless: only scan over envelope metadata (§6.4)
  index_not_ready,   # lexical or semantic projection missing/building/stale (§6.3)
  no_semantic_index, # collection embed_policy = off, or no embedding provider
  no_graph,          # 0040 unavailable in this deployment
  version_unavailable# requested as_of/history not retained for this collection (§4.4)
}
```

The caller sees which rungs ran (`engine.rungs_available` + `index_status`) and why it was degraded, so
Studio can badge "results may be incomplete - semantic index building" rather than present a partial
result as complete.

### 8.4 Signature-stability guarantee

The request/response signature in §8.1-§8.2 is the contract. **Engine swaps never change it.** Adding a
rung, upgrading Tantivy or the ANN engine, changing `AcceleratorPolicy`, moving embedding from local to
a keyed worker, or switching a deployment from keyed to blind changes only `engine`/`index_status`/
`degraded`/`match_via`/`reduced` *values* - never the field set, never the tool name, never the meaning
of an existing field. New capability is added as additive optional fields (the 0033 `FieldType`
extension pattern), never as a breaking change. This is the property that lets Studio (0061 profiles)
stay entirely on Tier-1 and remain oblivious to Tier-2 (see the Tier-1/Tier-2 audit, _QUEUE8).

## 9. Edge Ports: One Index, Two Faces

**DP8 - decided: the Tier-2 compatibility ports are egress skins over the *same* indexes the internal
ladder uses. No forked second copy.**

The external compatibility ports (0061 §10.3) present the same accelerators outward that
`substrate_search` uses inward:

| External port | Backed by | Same index as |
| --- | --- | --- |
| OpenSearch-compatible served profile (_QUEUE2 280b-280h, over `fts`) | 0033 Tantivy engine | the **lexical rung** of this ladder |
| Qdrant / Pinecone compatible vector ports (0017 RD12; Qdrant REST/gRPC + Pinecone REST source-backed) | 0017 vector layer | the **semantic rung** of this ladder |

Shared-index contract (normative, so the Tier-2 build does not fork a second copy):

- **One index per accelerator, two faces.** The lexical Tantivy index that backs `substrate_search`
  lexical mode is the *same* index the OpenSearch port serves. The 0017 vector set + ANN accelerator
  that backs semantic mode is the *same* set the Qdrant/Pinecone ports serve. There is exactly one
  build, one derived-artifact-store entry (0033 N.6 / 0017 §5), one rebuild path per accelerator.
- **Ports are presentations, never the base model.** The external protocol shape (OpenSearch DSL, Qdrant
  points/filters) is a wire projection; it never becomes the storage model and never becomes a second
  source of truth (_QUEUE2's founding rule: a presentation must not become the base model). Writes
  through a port land in Loom source state and become searchable after refresh (0033 N.10); the derived
  index stays identity-excluded.
- **Same freshness + auth.** Ports observe the same `INDEX_NOT_READY`/rebuild lifecycle and the same PEP
  authority (read-only security shims, _QUEUE2 280f); OpenSearch-shaped security mutation is rejected,
  Loom auth/ACL/audit remain authoritative.
- **Ladder unification obligation.** The Tier-2 tasks (_QUEUE2 280b + the vector ports) MUST build
  against the accelerators this ladder consumes. If a port needs an index shape the ladder does not
  already build, the ladder's build is extended once and both faces consume it - the port never
  maintains a private copy.

This is the whole "Tier-2 at the edges" plan (0061 §10.3): foundation = shared rebuildable accelerators
behind the Tier-1 interface; edges = compatibility presentations projecting those same indexes outward.

## 10. CLI and Tool Surface

```text
loom search <store> <query> [--workspace <ns>] [--collection <c>] [--field <name>]
                          [--limit N] [--offset N] [--format text|json]

target:
loom search <store> <query> [--mode text|regex|semantic|hybrid|similar_to] [--similar-to <entity>]
                    [--workspace <ns>] [--collection <c>] [--facet <f>]
                    [--as-of <root|seq> | --history] [--limit N] [--offset N]
                    [--rrf-k N] [--format text|json]
loom fts    ...    # unchanged: collection-scoped lexical surface (0033 decision 4)
```

- `loom search` with no scope flags = whole-Loom hybrid, degrading down the ladder (§5), permission
  pre-filtered (§7). Scope flags narrow breadth (§2).
- `substrate_search` is the dotted MCP/tool spelling of the same interface (flattened per 0043 naming);
  `loom fts` remains the separate collection-scoped lexical surface and is intentionally *not* an alias
  for `loom search` (0033 decision 4, _QUEUE2 batch 280).
- `similar_to` has no free-text query; `--similar-to <entity>` seeds the semantic rung (§5).

## 11. Conformance and Tests

- **RRF determinism vectors.** Fixed lexical + semantic rank lists -> fixed fused order, including the
  §4.3 tie-break chain; identical output across engine presence/absence and shard layout.
- **`match_via` correctness.** Single-rung hits tagged by rung; multi-rung hits tagged `hybrid` with
  `contributing_rungs`; graph hits carry `via_edge`/`via_entity`.
- **Version-scope per rung.** `latest`/`as_of`/`history` behavior per §4.4, including `INDEX_NOT_READY`
  and scan fallback for `as_of`, and `version_unavailable` degraded reason where retention is absent.
- **Degraded-mode signalling.** Keyless deployment -> scan-only + `degraded.reason = no_keys`, never
  empty; missing semantic projection -> hybrid degrades to lexical with `index_status.semantic` reported.
- **Permission pre-filter.** Forbidden entities never appear in hits, counts, or affect RRF rank
  (existence-leak test); graph expansion cannot surface unauthorized entities.
- **Embedding lifecycle.** Enqueue-on-write does not block writes; coalescing rebuild; incremental
  re-embed touches only diff-changed entities; stale model identity forces rebuild before trust; blind
  host cannot embed (stores opaque vectors only).
- **Shared-index port parity.** The OpenSearch port and lexical rung return the same matches over the
  same index; the Qdrant/Pinecone ports and semantic rung return the same neighbors over the same vector
  set; a single rebuild updates both faces.
- **Signature stability.** A golden request/response schema test that fails on any breaking change to
  §8.1-§8.2; adding a rung changes only values, not the schema.
- No network access in ordinary CI (embedding uses a fake `TextEmbedding` fixture; hosted-provider paths
  behind ignored/opt-in recipes).

## 12. Decision Points

### DP1 - New composition spec vs extend 0033

**Question:** Does unified search / whole-Loom `loom search` get its own composition spec or extend 0033?

**Decision:** New composition spec **0064** (the provisional "0062" from the _QUEUE8 DP was already
assigned to Inference Model Downloads; 0064 is the next free number after 0063). Capability
`unified-search`. Mirrors the 0040 precedent: a capability that composes facets, not a new facet. 0033
stays the lexical FTS facet; 0017 stays the vector facet; 0040 stays the graph composition; 0061 §10
pins the interface + decoupling rule and points here for the deep contract.

**Consequence of deferring:** the ladder/fusion contract has no owning spec; Studio's decoupling rule
(0061 §10.3) points at a spec that does not exist.

### DP2 - Fusion method

**Question:** RRF-only, or also expose score-normalized/weighted fusion?

**Decision:** RRF-only, rank-based, default `k=60` (per-query override). No cross-engine score
normalization. Raw per-rung scores returned for display only. One deterministic fusion path (§4.1).

### DP3 - Tie-breaks

**Question:** Authoritative deterministic tie-break chain when fused scores tie?

**Decision:** `match_via` priority -> `entity_id` -> `as_of` root. Raw scores and recency are not
tie-break inputs (§4.3).

### DP4 - Embed content scope

**Question:** Which content_types get embedded by default?

**Decision:** Versioned bodies + append-only content, per-collection opt-out; body-revision embedding is
per-collection opt-in (§6.1).

### DP5 - Embed trigger

**Question:** On-write synchronous, batch, or hybrid?

**Decision:** Async keyed index-worker, enqueue-on-write, reusing the 0033/0017/0040 accelerator
lifecycle (`INDEX_NOT_READY`, coalescing, snapshot-stamped, incremental via 0003b/0040 diff). No forked
pipeline (§6.2). This is the answer the composed facet specs already imply.

### DP6 - Embedding provider posture

**Question:** Local in-process vs hosted embedding default?

**Decision:** Local in-process (candle/MLX; models via 0062) is the default for curated small models;
OpenAI-compatible HTTP is opt-in, never the silent default. Per-workspace model identity (0050); the
determinism spine keeps embedding output local-only, never a sync source of truth (§6.4).

### DP7 - Permission-scoped fusion

**Question:** Pre-filter per rung or post-filter after fusion?

**Decision:** Pre-filter per rung, then fuse. No post-filter stage; graph expansion honors the same
predicate (§7).

### DP8 - Edge-port index sharing

**Question:** Do the Tier-2 ports project the same indexes the internal ladder uses, or a separate copy?

**Decision:** Same indexes - one index, two faces. The Tier-2 build (_QUEUE2 280b + vector ports) MUST
build against the ladder's accelerators; a port never maintains a private copy (§9).
