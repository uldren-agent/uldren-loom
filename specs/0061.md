# 0061 - Operation Substrate

**Status:** Target design. **Version:** 0.1.0-target.
**Capability:** `substrate`.

**Depends on:** 0002 (object model), 0003d (cross-facet commit diff), 0016 (graph layer), 0020
(document layer), 0021a (structured append-log storage), 0023 (columnar layer), 0027/0028 (access
control), 0033 (search layer), 0035 (durable delivery), 0036 (locking and coordination), 0043 (MCP
serving surface). **Relates to:** 0006 (synchronization), 0029 (events and triggers), 0031
(end-to-end encrypted sync), 0011a (SQL facade; §14 plan-layer relationship),
`specs/studio/SLACKISH.md`, `specs/studio/DRIVEISH.md`, `specs/studio/JIRAISH.md`,
`specs/studio/PAGES.md`, `specs/studio/WEBISH.md`, `specs/studio/LIFECYCLE.md`,
`specs/studio/SURFACES.md`, `specs/studio/ADOPTION.md`.

This spec defines the shared operation substrate that the Studio application profiles assume but do
not own, plus the assistant-facing cross-facet ergonomic tools built on the same primitives. Each
Studio profile currently restates the same envelope, sequencer, cursor, order-token, conflict,
annotation, retention, and projection requirements with local vocabulary, and each lists the same
items as open decisions. This spec pins them once. Profiles shrink to: operation kinds, validation
rules, projection layouts, and tool vocabulary.

Reusable implementation lives in one crate, `loom-substrate`. Studio profiles are feature-gated
modules of that crate until a consumer outside the Studio family justifies a split.

Current source backs part of the human-facing surface contract through `loom-substrate::surfaces`.
The module provides canonical app-definition, elicitation request/response, prompt-handoff, and
render-frame records used by `specs/studio/SURFACES.md`. The MCP app resource and authoring
mechanism remains owned by 0043 and `loom-mcp`; browser rendering, iframe bridge execution, golden
visual fixtures, and profile-specific app implementations remain target work.

## 1. Naming

Specs write tool names in dotted canonical form (`search`, `tickets.transition`). The MCP
serving surface projects those names using 0043's flattened transport spelling
(`search`, `tickets_transition`) and keeps output schemas stable under workspace,
collection, and principal elision. The dotted spec name and flattened transport name identify the
same operation; the dotted spec name is not a temporary placeholder.
Current source backs the ticket transition contract through the existing MCP `tickets_update` tool
with `target_status`, optional observed audit fields, and attached `set_fields`. Validation uses the
ticket project's configured active workflow when present. Projects without an active workflow are
permissive by default: direct owner-directed status repair operations such as `planned` to
`accepted`, `planned` to `feedback_available`, and `blocked` to `accepted` are allowed and recorded
as normal ticket transitions. A dedicated `tickets_transition` projection remains target work, but
configured-workflow validation uses the same workflow-edge contract described by JIRAISH §7.3.

Substrate tools use the `substrate.` prefix for operation-substrate reads and writes:
`substrate_changes`, `substrate_refs`, `substrate_transact`, `substrate.view_*`. Store-wide search is
the top-level `search` tool because it is user-facing, not a substrate-only implementation detail.
Profile tools keep their domain
prefixes (`tickets.*`, `pages.*`, `chat.*`, `files.*`). The naming rule: if a user's sentence would
name the thing, it is a profile tool; if only an assistant's execution plan would name it, it is a
substrate tool. Search is the exception that prevents profile-specific duplication: profile content
publishes into the shared search substrate, and callers use `search` with scope/filter arguments
rather than separate tools such as `tickets_search` or `meetings_search`.

## 2. Operation Envelope

One canonical envelope for all profile operations. This supersedes per-profile open decision #1 in
SLACKISH §15, DRIVEISH §17, JIRAISH §18, and PAGES §19.

```text
operation envelope:
  workspace_id
  app_id            profile capability, e.g. tickets
  scope_id          channel, project, space, or folder
  operation_id
  operation_kind    profile-defined, workspaced string
  sequence          assigned by sequencer
  actor_principal
  actor_kind        user | agent | service
  timestamp_ms
  idempotency_key
  base_root         scope root the client built against
  base_entity_version optional
  target_entity_id  optional
  payload_digest
  payload           profile-defined, canonical CBOR
  policy_labels
  signature         optional
```

Agent-authored operations additionally carry the agent identity block shared by all profiles
(`agent_id`, `model_or_runtime`, `operator_principal`, `source_entity_ids`, `tool_calls`,
`confidence`, `policy_labels`, `trace_digest`).

The envelope encoding is canonical CBOR with pinned bytes, negative decode tests, and cross-language
vectors before it becomes identity-affecting.

## 3. Sequencer Contract

One sequencer protocol, blind-capable, parameterized by profile. This supersedes per-profile open
decision #2.

The sequencer: authorizes the principal by label, validates envelope shape without reading payload
plaintext, assigns the next scope sequence, allocates any requested aliases and order tokens,
persists the opaque operation, advances the scope root by compare-and-swap (0036 primitives), and
emits wakeups (0035 delivery, 0029 triggers). Replay is by sequence range per scope. The blind,
keyed-worker, and per-actor-log topologies described in the Studio profiles are deployment modes of
this one contract, not separate protocols.

**Envelope visibility rule** (owner decision 2026-07-04, from the SLACKISH session): a blind host
sees envelope metadata - ids, `operation_kind`, sequence, actor principal and kind, timestamps,
idempotency key, `base_root`/`base_entity_version`, payload digest and size, policy labels,
block-id-count metadata - and never the payload. `operation_kind` is deliberately metadata, not
payload: blind hosts shape-validate, rate-limit and meter per kind (§20), route notification wakeups
per kind (§16 decision 7), and enforce scope operating modes (§7.1) without keys, at the documented
cost that activity type and volume are visible to the host. SLACKISH §2.1 records the per-feature
ledger.

## 4. Stable Identity and Aliases

Entities have stable ids (`ticket_id`, `page_id`, `message_id`, `file_id`). Human-readable keys
(`PROJ-123`, page URLs, paths) are mutable scope-local aliases allocated by the sequencer. Renaming a
scope or entity rewrites aliases, never identity. The allocator contract (uniqueness, collision
rules, rename semantics) is owned here; profiles define alias formats. How aliases appear in free
text and bind to stable ids is the cross-profile reference grammar (§19).

## 5. Order Tokens

One order-maintenance library for JIRAISH rank (§7.4), PAGES page-tree order (§8.2), and any
future ordered collection: sparse tokens, insert-between allocation, deterministic tie-breaks for
concurrent inserts in the same gap, periodic compaction that does not change entity identity, and
sorting by `(order_token, entity_id)`.

Current source backs this as fixed-width 128-bit sparse tokens encoded as 32 uppercase hexadecimal
characters. Lexicographic token order matches numeric order; concurrent inserts in the same gap use
`(token, entity_id)` as the deterministic display key; compaction re-spaces tokens in current display
order without changing entity ids. Exhausted gaps return `CONFLICT` and require compaction before the
next insert in that gap.

## 6. Guarded Writes and Conflict Records

Any operation may carry `base_entity_version`. If the entity advanced past the base, the sequencer
applies the profile's conflict policy: merge disjoint field updates, sequence-and-record for same
field scalar conflicts, revalidate for state-machine transitions, or reject. Recorded conflicts use
one schema:

```text
conflict record:
  conflict_id
  entity_id
  base_version
  losing_operation_id
  winning_operation_id
  field_or_region optional
  resolution_state open | resolved | superseded
```

Conflict records are visible in entity history and resolvable by later operations. JIRAISH field
conflicts (§7.2) and PAGES page-save conflicts (§8.1) are instances.

Current source backs the conflict record schema and guarded-write policy matrix in `loom-substrate`.
Conflict records are canonical CBOR arrays under `loom.substrate.conflict.v1` with deterministic
conflict ids derived from the entity, base version, losing operation, winning operation, and optional
field or region. The policy evaluator source-backs merge-disjoint-fields, record-scalar-conflict,
revalidate-state-machine, and reject-stale-base outcomes without mutating state.

## 7. Annotations

Comments, messages, reactions, and inline comments across all four Studio profiles share one shape
and are one subsystem here:

- creation appends in sequence order;
- edit is a later operation preserving version history;
- redaction is a later operation that hides visible content while the audit fact persists;
- hard deletion happens only through retention policy (key retirement, live roots, GC);
- reactions are add/remove annotation pairs;
- anchors are typed: `entity` (issue comment, file comment), `thread` (chat), `range` (inline page
  comment, with re-anchor-on-edit and stale-anchor marking owned by the §9.1 body model).

Operation kinds: `annotation.added`, `annotation.edited`, `annotation.redacted`,
`annotation.resolved`, `annotation.reaction_added`, `annotation.reaction_removed`,
`annotation.pinned`, `annotation.unpinned` (owner 2026-07-04, SLACKISH session: pinning is
scope-scoped, sequenced, auditable, and cross-profile - chat pins, starred issues, and favorited
pages are one kind).

Reaction kinds are arbitrary strings validated against an organization **emoji registry** - scope-level
data like label taxonomies (§7.1), auditable and importable; unicode emoji are implicitly registered
(owner 2026-07-04, SLACKISH session).

Current source backs the shared annotation event schema and deterministic derived annotation store in
`loom-substrate`. Source-backed operations are add, edit, redact, resolve, reaction add/remove, and
pin/unpin over typed entity, thread, and range anchors. Redaction preserves the annotation id and
event history while clearing visible body content and recording redaction metadata; later edits to a
redacted annotation return `CONFLICT`. Source also backs the canonical emoji registry model:
organization custom reaction kinds are sorted and deduplicated, Unicode emoji sequences are implicitly
accepted without registry storage, and `loom-conformance` pins the registry bytes in the
`substrate-model` runner. The registry is stored at a reserved substrate path scoped by workspace
profile id, and MCP source-backs `chat_emoji_list`, `chat_emoji_register`, and
`chat_emoji_unregister`. MCP and hosted Chat reaction-add paths validate custom reaction kinds
against the same stored registry; reaction removal remains allowed for previously registered kinds.
Import hooks, hosted registry-management routes, attachment-backed custom emoji images, and profile
UX remain target work.

### 7.1 Other Shared Facilities

The annotation audit generalizes: several more "profile features" appear in all four Studio profiles
with identical semantics and belong here, not in each profile. Profiles keep their domain tool doors;
the model and semantics live in the substrate.

- **Attachments.** SLACKISH §6, JIRAISH §9, PAGES §10, and DRIVEISH define the byte-identical
  schema (`attachment_id, digest, name, media_type, size, uploaded_by, created_at_ms, scan_status,
  retention_class`). One substrate model: content-addressed blob or chunk list, metadata object,
  reference from any operation, scan/preview/retention worker behavior. Any entity - issue, page,
  message, roadmap item, structure node, lifecycle stage record - can carry attachments; profiles do
  not re-specify this.
- **Mentions.** Free-text references to principals and entities, extracted per the cross-profile
  reference grammar (§19) into typed edges at sequence time. Feeds notifications and
  `substrate_refs`.
- **Watches.** A per-principal subscription registry (watch an entity, scope, label, or query) with
  identical add/remove semantics everywhere. Watches are the input to the notification projection
  (`ADOPTION.md` G3); profiles stop defining their own watch models.
- **Labels.** Concurrent adds merge; removal is sequenced and auditable (semantics already identical
  in JIRAISH and PAGES). Label taxonomies are scope-level data.
- **Saved queries.** Named, shareable predicate trees (§14). JIRAISH filters and PAGES
  search filters are instances; saved queries are also valid view sources (§8) and watch targets.
- **Field definitions** (owner decision 2026-07-04, Notion/Asana session). The typed custom-field
  system pinned in JIRAISH §20.1 - closed primitive set (plus `list<T>`, the `opaque_json` escape,
  and `date_range` added 2026-07-04), organization-global definitions with per-scope contexts,
  stable-id enum options with retirement, pinned type widenings, `imported_computed` freezing, and
  deterministic post-sequence validation against §14 predicate literals - is a substrate shared
  facility. JIRAISH issue custom fields and PAGES database structures (PAGES §21)
  consume one type system with one validation path and one vector set; JIRAISH §20.1 remains the
  defining text (same pattern as the §9.1 body model with PAGES §5).
- **Scope operating modes** (owner decision 2026-07-04, from the JIRAISH import session). Every
  scope (project, space, channel, folder) has an admin-set operating mode enforced at the policy
  enforcement point and sequencer: `read_write` (normal), `read_only` (all reads, writes rejected
  with a policy message), `mirror(source)` (read-only plus an incremental one-way import bridge
  feeding it - JIRAISH §25.4 and the other profiles' coexistence bridges are instances). Mode
  changes are audited operations; `mirror -> read_write` is the cutover operation, which stops the
  bridge and records the last-synced baseline.

Current source backs the shared-facility model layer in `loom-substrate::facilities`: attachment
metadata, watch registries, canonical label sets, scope operating-mode records, saved queries
embedding the §14 predicate identity, mentions, field definitions, the full shared field-value model,
field-value validation, safe field-type widening checks, and JSON projection helpers. The
`substrate-model` conformance runner covers field-definition canonical round-trips, field-value
canonical round-trips, safe widenings, and validation mismatch rejection. These are canonical
reusable models and tests; field-definition storage facades, MCP tools, projection workers, and
profile-specific UX remain target work.

## 8. Projections and Materialized Views

The operation log is the source of truth; everything user-visible is a projection. A projection is a
named, versioned, deterministic function from source roots to a derived artifact, registered with the
engine and maintained by the projection worker:

```text
view definition:
  view_id
  source scopes and facets
  projection function reference (versioned)
  output facet or media type
  freshness policy  on-write | on-read | scheduled
  output digest + source digests (0003d model)
```

Views are served as `loom://` resources. Profile projections (issue tables, board views, page trees,
backlink graphs, unread counts) and assistant-facing artifacts (a generated markdown mirror of a
planning store) are the same primitive with different projection functions.

Current source backs the reusable view-definition contract in `loom-substrate::view`: canonical
view-definition encoding, deterministic source scope/facet/digest normalization, freshness policies,
and a deterministic registry model. MCP `substrate_view_define` persists canonical definitions under
the reserved `.loom/substrate/views` registry path after explicit PEP authorization,
`substrate_view_get` reads them back through the same PEP boundary, and `substrate_view_list` lists
the registered definitions for a workspace. View definitions now capture the workspace's current
branch tip as `source_digests` when registered, so callers can tell which committed source root a
definition was recorded against. `substrate_view_get` also executes source-backed built-in
projection references for tickets open-item views, the general planning-store markdown mirror,
Meetings extraction-review views, and lifecycle operation-log views; unknown projection references
return an explicit target-status payload instead of fabricating output. The standard Studio status
view resource is source-backed at the MCP resource layer and reports source-backed sections for VCS
status, open merge conflicts, pending ask decisions, suggested prompts, ticket-backed assigned/open
items from the reserved `studio` organization, a deterministic `text/markdown` status body from the
same ticket list, and compact `substrate_changes` summaries over ticket and page operation cursors
for the same organization. The status view also source-backs an active-lifecycle section from the
canonical lifecycle operation log when present, and a review/comment-ownership section from the
Meetings extraction-review projection when a Meetings profile snapshot exists. Generic unresolved
inline-comment ownership and projection workers for arbitrary projection functions remain target
work.

Studio indexing has three source-backed enqueue paths. First, operators can run
`loom studio reindex <store> <workspace> [--profile <profile>]` to persist a Studio
embedding-projection job. Second, MCP clients can call `studio_reindex` with `workspace` and an
optional `profile` to persist the same job through the MCP write surface. Third,
`loom vector workspace configure` and
`loom inference instance update` persist affected Studio embedding-projection jobs when the vector or
inference configuration changes. Missing built-in inference is persisted as `no_engine`, not treated
as a terminal skip. The current job identity is the Studio scope, source root, and embedding model
stamp, so retries coalesce. CLI `loom studio reindex` drains Meetings vector projection outputs into
physical vector records when a text-embedding instance is bound, removes invalidated or
retained-metadata vector records, and marks projection-output-level jobs ready. Per-request local MCP
`studio_reindex` follows the same drain path when the configured text-embedding instance is available.
Hosted runtime execution and Meeting-specific retrieval through vector/search indexes remain target
work.

**Standard named views.** The substrate defines a small set of views every deployment serves. The
first is the per-principal Studio status bootstrap view (`ADOPTION.md` G5):

```text
loom://{workspace}/studio/views/status/principal/{principal} ->
  active lifecycle instances and their stages (LIFECYCLE.md)
  assigned open items (issues, review requests, unresolved inline comments)
  changes since the principal's cursor (summarized)
  open conflict records awaiting the principal
  pending decision points and elicitations
  suggested stage-appropriate prompts
```

A fresh assistant session reads this one resource and has working context without chat history. The
workspace segment and the `principal/{principal}` suffix may be elided by an MCP session that is
already bound to those values; the post-elision URI is `loom://studio/views/status`, not a top-level
principal resource. The current source-backed resource returns workspace and principal identity,
source-backed VCS/conflict/decision/prompt sections, ticket-backed assigned/open items from the
reserved `studio` organization, a deterministic `text/markdown` status body from the same ticket list,
compact ticket/page operation-log change summaries, lifecycle operation-log summaries, Meetings
extraction-review ownership summaries, and a target marker for generic inline-comment ownership. The
general markdown mirror of a planning store is the second standard view and is source-backed through
`substrate_view_get` as `view:planning.markdown`.

## 9. Entity Versioning and Revisions

Editing a page five times must yield five viewable revisions. The substrate answers "commits, rows,
or hybrid?" with a hybrid that assigns each mechanism one strict role:

1. **Revision bodies live in CAS.** Every edit operation references its payload by digest. Large
   bodies (pages, spec documents, issue descriptions) are content-addressed snapshots or body-model
   deltas; viewing revision 3 is a CAS get by digest. Structural sharing makes retained revisions
   cheap. Every body revision carries a `content_type` (markdown, ADF, or the §9.1 canonical
   body model); bodies are stored native, per-type renderer/extractor registries handle display,
   search indexing, and mention extraction, and conversion between types is an explicit revision -
   pinned 2026-07-04 in JIRAISH §20.2, semantics shared by all profiles.
2. **Revision identity lives in the operation log.** The edit *fact* - actor, timestamp, base
   version, payload digest, sequence - is the operation itself. An entity's version number is the
   count of sequenced operations targeting it: `(entity_id, version_n)` resolves to exactly one
   operation. This is what "revision history" means in every profile.
3. **Revision tables are projections.** The "SQL table with 5 rows" exists, but as a derived,
   rebuildable index `(entity_id, version, sequence, actor, timestamp_ms, payload_digest,
   change_kind)` maintained by the projection worker. Profile history tools (`issues` history,
   `pages` list-history) and the document viewer's revision slider all read this one index.

Workspace/VCS commits are deliberately **not** the profile-visible revision identity. Commits are
the storage, sync, and audit substrate the operations and projections live in - but a commit batches
unrelated writes across entities, storage compaction may rewrite it, and blind replicas replicate
operations, not plaintext diffs. Commit history (0003b) and cross-facet diffs (0003d) remain the
history surface for *raw facet* data outside any profile; `substrate_changes` (§12) bridges both
worlds.

Scalar-heavy entities (ticket fields) do not snapshot the whole entity per edit: their history *is*
the operation list, and "as of version N" reconstructs by replay from the nearest checkpoint. The
sequencer emits a checkpoint operation every N edits as a replay bound. Body-heavy entities always
carry a digest per revision. Diffing two revisions uses 0003d for facet-shaped payloads or the
profile's body-model diff for rich documents.

Redaction and retention interact with revisions by retiring payload keys (crypto-shredding a
revision's content), never by renumbering: the operation fact and version number persist for audit
even when the body is gone.

Current source backs the reusable revision model in `loom-substrate::versioning`: CAS body
references, monotonic per-entity revision records, revision logs with latest/revision/root lookup,
checkpoint records, deterministic revision indexes that project history, latest, as-of-root, and
nearest-checkpoint lookups, and an atomic generic profile transaction state that compares expected
roots, validates expected entity revisions, appends revision rows, creates checkpoints, advances the
profile root, and leaves state unchanged on conflict. The same module source-backs idempotent
current-state backfill for entities missing revision-index rows. The CLI `loom studio revisions
rebuild` command uses that backfill path for Meetings snapshots, Drive metadata operation logs,
Pages operation logs, and Lifecycle operation logs, writing missing revision rows and checkpoints
without duplicating existing rows. MCP page publishing uses the generic transaction helper to
maintain the reserved
`.loom/substrate/revisions/{workspace_id}.lri` revision index for successful page publishes, so
published pages appear in `substrate_history` as `page:{page_id}` revisions with checkpoints.
Ticket creation and field updates use the same helper to maintain the reserved revision index for
`ticket:{ticket_id}` revision rows with checkpoints. Ticket ids are Loom-generated UUIDs; project
keys and external source identities resolve to that immutable id. MCP chat message create, edit, and
redact operations use the same helper to maintain the reserved revision index for
`chat:{channel_id}:message:{message_id}` revision rows with checkpoints. Hosted REST and JSON-RPC
chat message create and edit routes use the same typed Chat write surface and are covered by
protocol conformance against the same revision projection. MCP Drive committed uploads maintain the
same reserved revision index for `drive:file:{file_id}` file-content revision rows with checkpoints
through the generic transaction helper. Hosted REST and JSON-RPC Drive committed-upload routes use
the same typed Drive write surface and are covered by protocol conformance against the same revision
projection. `loom meetings import` maintains `meeting:{meeting_id}` revision rows with checkpoints
for changed imported meeting records through the same helper. Drive metadata operations, Meetings
review writes, Lifecycle definition and instance writes, and Pages space, draft-page, and structure
operations now maintain profile-specific revision rows. Storage placement beyond this reserved path,
profile-specific body deltas, and future profile-specific public tools remain target work.

### 9.1 Rich Body Model and Body Deltas

In §9 content_type terms (pinned later the same day by the JIRAISH session), this model is the
**canonical collaborative-editing content type**: markdown and imported ADF bodies are
non-canonical content types stored native per JIRAISH §20.2, and conversion into the canonical
type is an explicit revision. The two decisions were made independently and compose: §20.2 governs
typed bodies at rest; this section governs the one type that supports sequenced deltas.

Owner decision 2026-07-04 (PAGES body-model session): the rich body format and its delta
model are substrate-owned so every profile's rich text (page bodies, issue descriptions, rich
comments) is one format with one vector set. PAGES §5 is the primary consumer and defines
page-specific node kinds and import; the format itself lives here. The survey behind these
decisions (whole-page snapshots vs sequenced deltas vs CRDTs, scored against canonical bytes,
blind replica, crypto-shred, anchoring, import fidelity, and history size) is recorded in
PAGES §5.5.

**Model decision.** A pinned block-structured canonical document plus sequenced body deltas, with
snapshot revisions at publish boundaries. CRDTs are explicitly not adopted: the blind sequencer
(§3) already supplies a total order, which removes the property CRDTs exist to provide
(convergence without central order), while CRDT tombstone retention conflicts with per-revision
crypto-shred and the canonical-bytes rule (the conformance surface of a CRDT is the whole
algorithm, cross-language). Whole-page replacement (`body_replaced`) is the degenerate delta, so
snapshot-plus-conflict-record remains the no-collab and actor-log-topology mode. If a future
actor-log deployment needs offline auto-merge of bodies, a CRDT body (Loro-class, with shallow
snapshots) is the documented option, with its shred and canonical-bytes conflicts to be resolved
then, not now.

**Block document.** A body is a tree of blocks:

```text
body:    schema_version, root block list
block:   block_id, kind, attrs, children, content
content: text runs - each run is (text, canonically ordered mark set)
```

Substrate block kinds: `paragraph`, `heading(level)`, `list_item(list_kind)`, `code_block(language)`,
`quote`, `table` / `table_row` / `table_cell`, `divider`, `embed`, and `block_ref(entity_id,
block_id optional, mode block | section, pin optional)` - committed 2026-07-04: render another
block's content by reference (Notion synced blocks / Obsidian embeds), single source of truth,
read-through at render time. When `block_id` is absent the whole target body is transcluded;
`mode: section` (owner decision 2026-07-04; valid only when the target is a heading) renders the
heading plus its following sibling blocks up to the next heading of the same or higher level -
Obsidian `![[Page#Heading]]` embeds. Semantics pinned 2026-07-04 (owner decisions, Notion/Asana
session; resolves §16 decision 8):

- **No content caching.** A block_ref never stores target content. When the target is
  unresolvable - block removed, entity gone, epoch key retired (§9 crypto-shred), or the reader
  lacks grants - it renders a **stale-ref placeholder** naming the target by alias lookup where
  permitted ("from *Payment Runbook*, content unavailable"). A cached fallback would let
  transclusion defeat crypto-shred; degradation deliberately mirrors stale anchors.
- **Pinned revisions.** `pin` fixes the target at a specific revision: it renders while that
  revision is retained and degrades to the placeholder once shredded. Unpinned refs render the
  target head.
- **Cycles and depth: render-time enforcement.** References are payload-resident (§19.3), so a
  blind sequencer cannot see them - write-time cycle rejection is impossible in the blind topology
  and is not part of the contract. The renderer keeps a visited `(entity_id, block_id)` set and
  cuts revisits with a cycle marker; render depth caps at **8** nested transclusions (pinned
  default; conformance vectors use it). Keyed deployments may warn at write time as UX only, per
  the keyed-pre-validation-is-UX rule.
- **Search indexing: once, at the source.** Transcluded content indexes only where it lives;
  embedding documents get no duplicate index entries. "Which documents transclude this" is the
  backlink index; search surfaces may join at query time. Index size stays independent of embed
  count, and retention key-retirement purges exactly one entry (§10.1).
- **Backlinks.** Creating or removing a block_ref emits a typed `transcludes` edge into §11
  (payload-resident, keyed projection). This is an explicit structural act by the actor, not
  free-text extraction, so a semantic edge kind is correct - §19.2's extraction-never-infers rule
  is unaffected.

There are no list-wrapper nodes:
list items are blocks whose grouping is a render concern (Notion-style), which keeps normalization
local. Substrate marks: `bold`, `italic`, `underline`, `strikethrough`, `code`, `link(href)`.
Profiles register additional block kinds and marks (e.g. PAGES `macro_reference`,
`mention`, `status`); unknown kinds and marks are preserved as opaque canonical CBOR and
round-tripped untouched (unknown-node preservation).

**Canonical normalization** (applied after every delta; identity-affecting): adjacent runs with
identical mark sets merge; empty runs are forbidden; marks sort by kind string then canonical attr
bytes; attrs sort by key; sibling blocks sort by `(order_token, block_id)` (§5 order tokens govern
sibling order); empty non-void blocks are preserved (an empty paragraph is content), but zero-run
content arrays are encoded canonically empty. The body encoding is canonical CBOR with pinned
bytes, negative decode tests, and cross-language conformance vectors per the canonical-bytes rule.

**Offsets.** All text offsets (deltas, anchors) are UTF-8 byte offsets into a block's
concatenated run text and must fall on character boundaries; applying an offset off a boundary is
a validation error. Non-Rust clients count bytes, not chars - a documented conformance
requirement.

**Unicode policy (owner decision, 2026-07-04): text is stored verbatim.** No Unicode
normalization is ever applied on write: identity is over code points exactly as written, offsets
are over verbatim bytes, and imports round-trip byte-exact. The cost is deliberately pushed to
read time: the §10 search projection must fold (NFC + case) at index and query time so visually
identical text matches regardless of source form. Clients must not normalize text before
submitting deltas - a client that NFC-normalizes silently would corrupt offsets computed by
others against the verbatim bytes.

**Block identity.** Both allocation modes are supported; sequencer-allocated is preferred.
Preferred: the envelope requests a block-id count and ids derive deterministically from
`(entity_id, sequence, index)` - collision-free by construction, no registry, blind-safe; the
payload references new blocks by allocation index. Supported: client-minted 128-bit random ids
(offline drafts, import) declared in an envelope metadata field `introduces_block_ids` so the
blind sequencer can uniqueness-check them against its per-scope id set without reading the
payload.

Current source backs the canonical body model in `loom-substrate::body`: marks, text runs, ordered
blocks, block references, opaque profile extension blocks, canonical run merging, sibling ordering by
order token plus block id, canonical encoding and decoding, UTF-8 byte-offset range validation, body
patch stale-base rejection, insert/remove/move/set/splice/replace primitives, splice anchor
remapping, and epoch render gating for crypto-shred visibility. The body renderer source-backs
render-time `block_ref` read-through without content caching, missing-target degradation,
missing-block degradation, shredded placeholder degradation, cycle detection, and the default depth
cap of 8 nested transclusions. Page publish projects canonical `block_ref` targets into
`substrate_refs` as `transcludes` edges. `pages_get` source-backs page-profile rendered text for
published and draft canonical bodies, resolving `page:<id>` `block_ref` targets against published
page revisions and returning structured render issues for missing targets, missing blocks, shredded
targets, cycles, and depth limits. Public `document.patch`, import-time fidelity fixtures, and
cross-language conformance vectors remain target work.

**Delta operation set** (minimal primitives; editor commands - split paragraph, toggle list -
compose primitives inside one atomic operation):

```text
insert_block(block_id, parent_id, order_token, block payload)
remove_block(block_id)                    removes the subtree
move_block(block_id, new_parent_id, new_order_token)
set_block_attrs(block_id, attr patch)
splice_text(block_id, start, end, replacement runs)
set_marks(block_id, start, end, marks added, marks removed)
body_replaced(new body)                   degenerate whole-body replacement
```

Each primitive has pinned apply semantics, pinned normalization, and a pinned **offset-mapping
function** (how positions before/inside/after the edit map through it). One mapping definition
serves both client rebase and anchor re-anchoring.

**Concurrency protocol: reject stale base, client rebases.** A body-delta operation carries
`base_entity_version` (§6). The sequencer - blind, checking envelope metadata only - accepts the
delta only if the base equals the entity head; otherwise it rejects, and the client maps its
unconfirmed primitives through the intervening deltas (the offset-mapping function) and resubmits.
This is the ProseMirror collab protocol over the §3 sequencer. Consequence: **persisted body
history is strictly linear** - every stored delta applies cleanly to its predecessor, replay needs
no transform logic, and no transform function ever becomes identity-affecting. When a client
cannot rebase meaningfully (target block deleted), it surfaces the failure; publish-time merges of
divergent drafts fall back to the §6 conflict record, which remains the whole-page-mode conflict
path.

**Snapshots, epochs, and crypto-shred.** Publishing creates a snapshot revision: the full
canonical body in CAS, closing the **epoch** of deltas since the previous snapshot. Snapshot and
epoch deltas are encrypted under one epoch key. Retention hard-deletes revision N by retiring
epoch key N: the snapshot and the deltas that produced it become unreadable while operation facts
and version numbers persist for audit (§9). The previous snapshot is the replay bound - §9
checkpoints and epochs are the same mechanism for body-heavy entities. Deltas older than the last
retained snapshot are compactable per retention policy.

**Range anchors.** An inline anchor is `(block_id, start, end, base_revision, quoted_text)` -
offsets in UTF-8 bytes, plus the quoted text at creation time. Live re-anchoring maps the range
deterministically through subsequent deltas with the same offset-mapping function. When mapping is
lost or ambiguous (block removed, epoch shredded, range destroyed), the anchor degrades to
block-level or page-level with a stale-anchor marker, and the quote still renders the annotation
meaningfully and feeds heuristic re-anchor assist. This subsection owns range-anchor semantics for
the §7 annotation subsystem.

**Conformance vectors:** canonical body encode/decode (positive and negative), normalization
idempotence, each primitive's apply + offset mapping, rebase round-trips, anchor mapping and
degradation, epoch shred (operation facts survive, payloads unreadable), and cross-language byte
equality.

## 10. Cross-Facet Search

`search` builds on the 0033 search facade; it does not define a new engine. The substrate
contributes auto-indexing: registered projections feed document, columnar, graph-property, ledger,
and annotation content into 0033 search collections as derived state. Query modes `text` and `regex`
ship first (0033's linear scan is an acceptable initial implementation behind the unchanging tool
contract); `semantic` waits on 0017/0040. Because body text is stored verbatim with no Unicode
normalization (§9.1), the search projection is required to fold text - NFC plus case folding - at
both index and query time, so "café" matches "café" whichever normalization form either was written
in. Hits are typed and context-enriched:

```text
search hit:
  facet, collection, entity_id, field
  snippet, offsets
  scope context: owning entity, status fields, refs (from graph projection)
  as_of root
```

Current source backs the reusable request/response contract in `loom-substrate::search`: modes,
latest/as_of/history version scopes, collection scope, typed hits, rung metadata, reduced responses,
and degraded signalling. CLI `loom search` and MCP `search` source-back the current store-wide
lexical scan baseline across readable FTS collections with optional workspace, collection, and field
filters. Historical execution, RRF fusion, semantic and graph rungs, and shared-index edge ports
remain in 0064 and downstream tasks.

### 10.1 Search and Versioning

Search indexes are projections, and projections track roots - so every query and every hit is
pinned to an explicit root. The tool takes a scope:

```text
scope:
  latest              default: the current root of each searched scope
  as_of(root | seq)   evaluate against a historical root
  history             match across all retained revisions and operations
```

Rules:

- **Latest** searches the index projection of the current root. The index worker updates on root
  advancement; a hit always reports the root it was computed against, so an app can detect and badge
  staleness rather than lie about it.
- **Append-only content is inherently historical.** Operations, annotations, ledger entries, and
  audit records never rewrite - their "latest" index *is* their full history. "Where did we discuss
  backpressure?" hits a redacted-but-audited comment in latest mode by design (subject to grants and
  redaction visibility).
- **Versioned bodies default to latest-only.** Pages, spec bodies, and descriptions index their
  current revision. Indexing retained prior revisions (`history` mode over bodies) is a per-collection
  index policy, because it multiplies index size and because retention key-retirement must be able to
  purge a shredded revision from the index - the retention worker owns index expiry for retired
  digests.
- **`as_of` search** evaluates against the revision set reachable from a historical root: either a
  retained index snapshot for that root, or a scan fallback. Same contract, implementation may differ
  by deployment.

The assistant-facing rule of thumb encoded in tool descriptions: "find X" -> latest; "did we ever /
where did we discuss" -> history; "what did the plan say at the release cut" -> as_of.

Worked query examples (keep these; they define the intended intuition):

```text
"find the spec that mentions backpressure"      -> latest over body indexes
"where did we discuss backpressure?"            -> latest over annotations/operations is
                                                   sufficient: append-only content is already
                                                   historical, including redacted-but-audited
                                                   comments (subject to grants)
"did we ever have a page on flow control?"      -> history over body revisions, available only
                                                   where the collection's index policy retains
                                                   prior revisions
"what did the plan say at the release cut?"     -> as_of(release root); baseline-vs-current is
                                                   this query at two roots
```

### 10.2 Engine Ladder (scan -> lexical -> semantic -> hybrid -> graph)

`search` is an **interface, not an engine.** Studio profiles and the assistant code to the
tool contract; the engine answering a query is a deployment/scale/key-posture choice that can change
without touching the contract or any caller. The ladder, each rung a rebuildable accelerator behind
the same contract (the 0017 vector philosophy - "exact search is the portable contract, accelerators
are optional derived state" - generalized to all of search):

| Rung | Engine | Backed by | Adds | Key posture |
| --- | --- | --- | --- | --- |
| Baseline | linear scan | 0033 base facade | always-available correctness, zero infra | runs degraded-blind over metadata; full text needs keys |
| Tier-1.5 | lexical / BM25 inverted index | 0033 + Tantivy (_QUEUE2 task 280b) | fast keyword/phrase/field FTS at scale | keyed worker or key-holding client (can't tokenize ciphertext) |
| Tier-1.5/2 | semantic / vector ANN | 0017 vector layer (HNSW/PQ; embeddings via 0050) | meaning-match - "backpressure" finds "flow control" | keyed only (can't embed ciphertext); blind host stores opaque vectors |
| Hybrid | lexical + vector fusion (RRF) | 0033 + 0017 | best single-query recall; the default when both indexes exist | keyed |
| Graph-expanded | retrieval over the reference/edge graph | 0040 GraphRAG (composes graph 0016 + vector 0017 + columnar 0023) | pulls related entities by typed edges/refs, RAG context assembly | keyed |

Contract additions to §10: query `mode` extends to `text | regex | semantic | hybrid | similar_to`
(`similar_to(entity_id)` = "find things like this ticket/page" - the find-related surface); every
hit carries a `match_via` provenance tag (`lexical | semantic | hybrid | graph`) and a relevance
score, so a caller can tell *why* something matched without knowing which engine ran. When multiple
rungs are available, `hybrid` is the default; when a deployment lacks keys, the engine transparently
falls **down** the ladder to scan/metadata and marks results degraded (never silently empty). None
of this changes the tool signature - it is all engine selection behind the interface.

The user scenario it answers: a ticket filed about "backpressure," later found by searching
"flow control" or "rate limiting" - semantic/hybrid mode over the 0017 embeddings, with graph
expansion surfacing the linked design page and decision. This is why `semantic` was reserved in §10
from the start; §10.2 fills in the ladder behind it.

### 10.3 Unified Search Interface and `loom search`

MCP `search` and CLI `loom search` are the same store-wide interface, sharing one engine ladder.
They search readable indexed material in one Loom, with optional workspace/collection filters. They
are distinct from the collection-scoped `loom fts` and MCP `fts_*` surfaces, which manage full-text
collections and documents. Studio profiles publish searchable projections into the shared substrate
and call the unified `search` interface when they need retrieval. Consequence: Studio is fully
decoupled from search. It codes to the interface and never learns whether a query was answered by
scan, Tantivy, vectors, or GraphRAG. Swapping or upgrading the engine is invisible to every Studio
profile and every Surface.

**Tier-2 at the edges, over shared indexes.** The external compatibility ports are egress skins over
the *same* accelerators this ladder uses, never separate copies: the OpenSearch-compatible port
(_QUEUE2 Tier-2, Tantivy-backed) exposes the same lexical index that backs `search`
lexical mode; the Qdrant/Pinecone-compatible vector ports expose the same 0017 index that backs
semantic mode. One index, two faces: internal (the Studio-facing interface) and external (the
wire-compatible port for outside clients, BI, notebooks). That is the whole "Tier-2 at the edges"
plan: foundation = shared rebuildable accelerators behind the Tier-1 interface; edges = compatibility
presentations projecting those same indexes outward.

The deep composition design (the whole-Loom `loom search` surface, the ladder's ranking/fusion
contract, embedding pipeline and freshness, and how it composes 0033/0017/0040) has its own
composition spec - **0064 (`unified-search`)**, analogous to 0040 GraphRAG being a composition, not a
facet. 0064 owns the RRF fusion contract, `match_via` provenance, deterministic tie-breaks, per-rung
`as_of`/`history` interaction, the embedding pipeline (content scope, async worker, keyed-only
constraint), permission-scoped pre-filter fusion, the decoupling request/response signature, and the
shared-index edge-port contract. This section pins only the interface and the decoupling rule; 0064 is
the deep contract. `loom search` is whole-Loom scope within one Loom: everything in the Loom, with an
optional workspace/collection/facet narrowing, not federation across multiple Looms.

## 14. Predicate Surface

Decision (owner, 2026-07-04): **option A - a structured expression tree is the only predicate
surface.** Query tools accept a versioned JSON expression tree at the MCP boundary, compiled to
canonical CBOR internally. The canonical CBOR form is what view definitions embed (it participates
in view identity/digests) and what the policy enforcement point inspects for field-level access
decisions - a tree walk, no string parsing.

Full expressiveness is pinned from day one (no staged grammar): boolean algebra, comparisons,
ranges, prefix/suffix/contains string matching, in-set, exists/null checks, and typed literals
validated against collection schemas. The version tag exists for canonical-encoding evolution, not
for delayed features.

The JSON shape is a deterministic AST:

```json
{
  "version": 1,
  "expr": {
    "op": "and",
    "args": [
      { "op": "eq", "path": ["status"], "value": { "type": "text", "value": "open" } },
      { "op": "gte", "path": ["priority"], "value": { "type": "i64", "value": 3 } }
    ]
  }
}
```

The root object has exactly `version` and `expr`. Version 1 accepts these node families:

| Family | Operators | Shape |
| --- | --- | --- |
| Constant | `true`, `false` | `{ "op": "true" }`, `{ "op": "false" }` |
| Boolean | `and`, `or`, `not` | `and`/`or` require non-empty `args`; `not` requires one `arg`. |
| Compare | `eq`, `ne`, `lt`, `lte`, `gt`, `gte` | Requires `path` and one typed `value`. |
| Range | `between` | Requires `path`, `lo`, `hi`, and optional inclusive flags defaulting to true. |
| Set | `in` | Requires `path` and non-empty typed `values`. |
| Presence | `exists`, `is_null` | Requires `path`; missing and null are distinct. |
| Text | `starts_with`, `ends_with`, `contains` | Requires `path` and a typed text `value`. |

`path` is a non-empty array of UTF-8 strings. Empty path segments, `.`/`..`, slash-bearing path
segments, and implementation-defined wildcards are invalid. A missing field is absent, not null:
`exists` is false, `is_null` is false, comparisons are false, and `not exists` is true. Text
operators are exact Unicode scalar comparison by default; case folding, tokenization, locale
collation, regex, and full-text ranking belong to search profiles, not the predicate identity
grammar.

Typed literals use this closed set in version 1:

| Type | JSON value | Canonical rule |
| --- | --- | --- |
| `null` | `null` | Only valid where a null value is meaningful. |
| `bool` | JSON boolean | Preserved as a boolean scalar. |
| `i64` | Decimal string or JSON integer in range | Canonicalized as signed 64-bit integer. |
| `u64` | Decimal string or JSON integer in range | Canonicalized as unsigned 64-bit integer. |
| `text` | JSON string | Preserved as UTF-8 text. |
| `bytes` | Lowercase hex string | Canonicalized as bytes. |
| `digest` | Loom digest string | Canonicalized as the digest byte representation. |
| `timestamp_ms` | Decimal string or JSON integer in range | Canonicalized as signed millisecond epoch. |
| `list` | Array of typed literals | Canonicalized recursively; mixed types are permitted only when the collection schema allows them. |

Floating-point literals are not part of predicate identity v1. Facets that need decimal or float
comparisons must lower through an owning schema profile that defines exact canonical comparison before
the predicate is admitted.

Canonical CBOR compilation emits a tagged pair:

```text
["loom.predicate.v1", expr]
```

where `expr` is the normalized AST using integer operator tags, canonical path arrays, and canonical
typed literal cells. JSON object member order, insignificant whitespace, decimal-string spellings,
and other presentation differences do not change the compiled bytes. Duplicate JSON keys, unknown
operators, unknown literal types, wrong arity, empty `and`/`or`, empty `in`, invalid paths, non-finite
or out-of-range numbers, invalid hex, non-canonical CBOR, and predicates beyond implementation
limits are rejected with `INVALID_ARGUMENT` before execution.

Implementation limits are part of the contract so hosted and binding implementations fail
consistently: maximum JSON input 64 KiB, maximum compiled predicate 32 KiB, maximum expression depth
32, maximum node count 1024, maximum `and`/`or` arity 256, maximum `in` values 1024, maximum path
segments 16, and maximum path segment length 128 bytes.

Conformance vectors for this surface must include:

- equivalent JSON spellings compiling to identical canonical bytes;
- every operator family with typed literal coverage;
- missing versus null behavior;
- schema-rejected mixed lists and float literals;
- negative decode cases for duplicate keys, unknown operators, wrong arity, invalid paths,
  over-limit predicates, non-canonical CBOR, and unknown predicate version tags;
- byte-for-byte identity for predicates embedded in view definitions and policy checks.

There is no SQL-string door. 0011a remains the human/SQL surface for tabular data and lowers to the
same plan layer, but SQL strings are never accepted where a predicate is part of a contract.
Byte-level CBOR parameters remain accepted for compatibility until their owning tools promote JSON
predicate inputs. The grammar and its canonical compilation are owned by this spec's conformance
vectors.

Current source backs the version-1 JSON predicate parser and canonical compiler in
`loom-substrate::predicate`: boolean nodes, comparisons, ranges, set membership, presence/null
checks, text operators, typed literals, path validation, depth/node/arity limits, canonical CBOR
encoding, and single-comparison lowering for current facet adapters. Current MCP projections use that
shared compiler for `columnar_select` and `kv_range`: columnar accepts the subset that lowers to its
existing single-column comparison filter, and KV accepts predicates over the typed `key` only because
KV values are opaque bytes. Full evaluator execution across every operator family, duplicate-key
decode vectors, and cross-language conformance vectors remain in 0025/this spec's conformance work.

### 15.1 Tool Surface Scaling

**Grouping is metadata, not a prefix (owner decision 2026-07-05).** Tools keep flat,
domain-prefixed spec names (`tickets_create`, `columnar_select`, `search`) and flattened
MCP transport names (`tickets_create`, `columnar_select`, `search`). Each carries a `group`
tag drawn from one shared taxonomy, surfaced identically in CLI help, MCP `tools/list`, and the
§15.1.5 discovery index. The taxonomy mirrors the CLI's existing `COMMAND_SECTIONS`
(`crates/loom-cli/src/main.rs`): Data Facets, PIM Facets, Compute, Versioning, Bindings, Security,
Management, General, plus a **Substrate** group (the §15 tools) and a **Studio** group (the
tickets/pages/chat/drive/web/lifecycle/surfaces profiles). The `studio` bundle capability
enables the whole Studio group at once. Names are never rooted under the group (`studio.tickets.*`,
`facet.columnar.*` were rejected because they diverge from the CLI and tax every identifier
forever). This is the "better than a filesystem" rule applied to tools: the grouping is queryable
metadata, not a string prefix.


Core facets plus five Studio profiles plus apps yields hundreds of tools; no client should ever see
them all. Strategies, in order of leverage:

1. **Capability gating (existing model, extended).** Tools register only for enabled capabilities -
   an organization without `chat` never lists `chat.*`. Area-scoped serving is already how the
   MCP host works (0043); profiles and the substrate follow it.
2. **Reads become resources, not tools.** Every `*.get`, `*.list`, and `*.history` has a `loom://`
   resource equivalent. Clients that speak resources need only write tools and search; this roughly
   halves the effective surface (0043 already serves resources).
3. **Lifecycle/stage-scoped surfacing.** The active lifecycle stage (LIFECYCLE.md) determines which
   tool groups are advertised: Ideate surfaces document and structure tools; Build surfaces issue
   and decision tools; Archive surfaces almost nothing. The local MCP host source-backs this as a
   session-bound active lifecycle context that filters `tools/list`, `get_tool`, and direct
   `tools/call`, with changes reported through MCP `tools/list_changed`. Tools remain callable when
   policy allows outside an active lifecycle context; surfacing is ergonomics, the PEP is enforcement.
4. **Shared facilities collapse duplicates.** One substrate model for attachments, watches, labels,
   annotations, and saved queries (§7.1) prevents five profile-specific copies of each (five
   `*.add_attachment` variants are ergonomics doors over one model. Deployments may expose only
   the substrate door plus the two or three highest-traffic domain doors).
5. **Complete tool discovery with client-side progressive loading.** The host advertises the complete
   visible tool inventory with callable schemas through standard `tools/list`. MCP clients that need
   progressive discovery own the context-loading policy; Loom does not expose a separate metadata
   search tool for schema lookup.

   Worked example: user says "move ISSUE-9 to Done." A client may choose to hold only a compact
   summary of the full inventory in model context, but the authoritative `tools/list` response still
   contains the complete schema for the matching ticket tool. Tool invocation remains a direct
   `tools/call` against the registered tool, and the PEP remains the enforcement boundary.

   Current source backs complete `tools/list` discovery for the current visible model-facing tool set.
   Current source also backs resource equivalents for stable
   substrate reads:
   `loom://{workspace}/substrate/views/{view_id}.json` maps to `substrate_view_get`, and
   `loom://{workspace}/substrate/refs/{target}.json` maps to `substrate_refs`; workspace-scoped MCP
   servers elide the workspace segment in those templates. `search` and
   `substrate_changes` carry explicit tool-only resource metadata because their query and cursor
   arguments do not identify one browsable resource.
   Current source also bounds model-facing payloads: collection-shaped MCP outputs default to 500
   items with `limit` and `offset` where ordering is stable, byte-heavy tool results and
   `resources/read` default to a 4 MiB delivered payload budget, and `fs_list_directory` pages the
   canonical `loom.fs.dir-listing.v1` payload without changing its result shape.
6. **MCP Tasks are cut from the current target surface.** Long-running tool work continues to use
   synchronous tools with progress and cancellation. Loom does not implement the legacy Tasks model,
   and it does not adopt the newer extension until a future queue can evaluate a stable extension and
   SDK contract.
7. **Prompts over micro-tools.** Compound flows (decompose a spec, close a sprint) ship as curated
   MCP prompts orchestrating existing tools rather than as dedicated tools.

## 19. Cross-Profile Reference Grammar

Decision (owner, 2026-07-04; resolves §16 open decision 6): one grammar for typed references across
all profiles. This is the connective tissue that makes ticket, page, message, and file one system
(ADOPTION G4); it feeds mentions (§7.1), the backlink index (§11), the notification projection
(§16 decision 7), PAGES structures, and import alias mapping (ADOPTION §1.3). This section
is appended after §18 deliberately so existing cross-references to §1-§18 stay valid.

### 19.1 Typed Reference Forms

Human-authored Studio text uses explicit sigils. Kinds are registered per profile capability; the
substrate reserves the core set:

```text
!ticket:LOOM-52      ticket by scope-local key alias
!ticket:018f...      ticket by stable UUID (always valid)
!page:substrate-predicates
!msg:...             message permalink alias or id
!file:...            file path alias or id
@name                principal mention (user, agent, or service principal)
#name                channel/scope mention
```

Bare titles, paths, ticket keys, and unprefixed `kind:id` text are never extracted from prose -
connectivity by fuzzy match is a search feature, not a reference. API fields that already carry a
target or entity context still use canonical `kind:id` values without the authoring `!` prefix.
Profiles may register additional kinds (`sprint:`, `dec:`, `structure:`) under the same rules; kind
names are lowercase and collision-checked at capability registration.

### 19.2 Mention Extraction

Extraction runs over free-text fields of an operation payload (bodies, comments, descriptions,
messages) and produces reference records, not text mutations:

- explicit `!kind:alias-or-id` forms, `@principal`, and `#channel` are extracted;
- code fences, inline code spans, and profile-declared verbatim regions are excluded;
- extraction is deterministic and versioned (a grammar version tag accompanies the references
  block) so re-extraction after grammar evolution is auditable;
- every extracted reference becomes a typed graph edge (`refers_to` for entity references,
  `mentions` for principals, `refers_to` scope edges for channels) feeding §7.1 and §11.

Extraction produces *reference* edges only. Semantic relations (`blocks`, `duplicates`,
`parent_of`) are explicit profile operations - free text saying "blocked by LOOM-52" creates a
`refers_to` edge; it does not infer a `blocks` relation. Assistants and apps may offer "make this a
blocking link?" as a follow-up elicitation, but the operation log never records a semantic relation
the actor did not state as an operation.

### 19.3 Binding and Resolution

Aliases resolve to stable ids at sequence time, and references bind to the ids, never the aliases
(§4):

- binding is performed by the key-holder - the client in the blind topology, the sequencer or a
  keyed worker in keyed deployments - against the operation's `base_root`, so the binding root is
  the same root the actor built against;
- the references block lives inside the encrypted payload, consistent with 0031: a blind host
  never sees the reference graph, and `substrate_refs` (§11) is a keyed projection;
- an alias that fails to resolve is sequenced anyway as `unresolved(kind, alias_text)`; a
  re-resolution worker binds it when the alias appears. Forward references ("LOOM-999", filed
  later) and out-of-order imports (a page citing issues whose import runs later - ADOPTION §1.3)
  are the normal cases, not errors. Unresolved references render as plain text until bound;
- writes never fail on reference hygiene: extraction and binding cannot reject an operation.

### 19.4 Serialization in Operation Payloads

The actor's text is preserved verbatim; references ride alongside it:

```text
references[] entry:
  span_start, span_end     byte offsets into the source text field
  field                    which payload field the span is in
  kind                     registered reference kind
  target_id                stable id, or absent while unresolved
  alias_text               the alias as written (unresolved key and audit record)
  alias_at_write           resolved display alias at binding time
  grammar_version
```

Payload digests, diffs, and audit therefore always match what the actor submitted. Late binding by
the re-resolution worker is a separate operation that records the binding; it does not rewrite the
original operation.

### 19.5 Rename Stability

References bind to stable ids; aliases are display (§4). Renaming a project (LOOM -> CORE) or moving
a page rewrites aliases, and every existing reference renders the current alias by id lookup -
nothing dangles. `alias_at_write` is retained so historical renderings and audit can show what the
actor actually saw. Alias reuse after rename follows the §4 allocator collision rules; a reference
bound before a reuse still points at the original entity, which is the reason binding happens at
sequence time against `base_root` rather than lazily at render time.

Current source backs the shared reference substrate in `loom-substrate::refs`: strict `kind:id`
canonical target parsing, deterministic extraction from `!kind:id`, `@handle`, and `#handle` text
tokens with byte spans, stable alias-binding records, canonical encoding for references, aliases,
sources, edges, alias indexes, and reference indexes, a deterministic reference index, and inbound-edge
lookup. MCP `substrate_alias_bind`,
`substrate_alias_release`, `substrate_alias_resolve`, and `substrate_alias_list` source-back active
scope-local alias management over the reserved alias index. MCP `substrate_refs` reads the reserved

candidate target catalog, queues, retry/failure records, and resolution records; `loom-tickets`

updates. A stateless
caller can run one bounded batch through `loom refs reconcile`; `loom refs status` reports pending,

## 20. Agent Budgets and Metering

Decision (owner, 2026-07-05, SLACKISH design session; resolves ADOPTION G7): agents are rate-limited
and metered independently of the principal who installed or invoked them. The design is
substrate-owned because the mechanism is identical for chat messages, issue edits, page writes, and
file operations; SLACKISH (§13, §2.1) is the first consumer. This section is appended after §19 so
existing §-references stay valid. Four pinned parts:

**Budgets (enforced at sequencer + PEP + 0015).** Budgets are per-agent-principal policy data -
admin-set, organization defaults with per-scope overrides - covering: operations per hour (total and
per `operation_kind`), automation compute (0015 metering units), storage bytes per day, and spend
where 0015 execution is billed. One budget document, three enforcement points, all consistent:

- the **sequencer** gates envelope-countable budgets (operation rates, bytes) *before* sequencing -
  blind-capable, because the §3 visibility rule makes actor, kind, and size metadata. A budget
  rejection carries a typed policy message, is audited, and is not a conflict record;
- the **policy enforcement point** applies the same budget document to reads and tool calls;
- **0015 execution** meters compute and spend during automation runs.

**Anomaly surfacing (pause + elicitation).** Distinct from budget exhaustion: an anomaly is a
metadata-only rate spike measured against the agent's own trailing baseline per scope ("agent posted
500 messages in 4 minutes"), detectable by a blind host. Budget exhaustion hard-rejects immediately
(the agent sees a typed error and can back off). An anomaly **pauses** the agent's further writes
and raises an elicitation to its `operator_principal` (fallback: scope admins): continue / kill /
raise budget. No answer within the policy TTL keeps the pause. Every pause, continue, and kill is an
audited operation.

**Kill switch (global kill + per-scope mute).** Both are audited operations enforced by principal id
in envelope metadata - blind-capable, effective on the next sequencing attempt in every topology.
Kill suspends the agent principal at the PEP and sequencer: envelopes refused, subscriptions
dropped, running 0015 executions cancelled. Grants remain intact, so reactivation is one audited
admin operation and the agent's history and audit trail stay queryable. Mute is the same mechanism
scoped to a single scope (channel, project, space, folder). Grant revocation remains authorization
design, not the emergency stop.

**Trace retention (`agent_forensics` class).** Every agent write must carry `trace_digest` (§2);
policy may waive it per low-stakes operation kind. Trace *content* (tool calls, prompts, model
output) is a sealed CAS payload - keyed access only - stored under the dedicated retention class
`agent_forensics` with a tenant-set window (default 90 days), legal-hold aware. Expiry
crypto-shreds trace content per §9 while the digest fact persists in the envelope forever: auditors
can prove a trace existed and was expired on schedule.

Profiles do not re-specify any of this; they cite it (SLACKISH §13). The notification projection
(§16 decision 7) consumes the same budget document for agent notification budgets.
